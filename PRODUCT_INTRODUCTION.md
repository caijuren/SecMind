# SecMind — AI-Augmented SOC Investigation OS

> 当告警如潮水般涌来，你需要的不是更多的仪表盘，而是一个能思考的安全搭档。

---

## 一、产品定位

**SecMind** 是一款 **AI 增强型安全调查操作系统（AI-Augmented SOC Investigation OS）**。

它不是 SIEM，不是 SOAR，也不是 AI 告警助手。它重新定义了安全运营的工作范式——

传统 SOC 的痛点在于：海量告警、人工研判、响应迟缓。分析师被困在"告警疲劳"中，真正的威胁往往淹没在噪音里。更致命的是，传统工具把 Alert、Incident、Investigation 混为一谈，导致分析师在原始信号和调查结论之间缺少结构化的跃迁路径。

SecMind 的核心创新在于建立了**三层流水线抽象**和**双轨研判机制**，将安全运营从"告警驱动"的被动响应，升级为**假设驱动的主动调查**：

```
Alert（原始信号）→ Incident（事件聚合）→ Investigation（假设推理）→ Decision（决策响应）
```

AI 不直接"结论化"，而是分层递进——每一层有明确的输入、输出和责任边界。AI 在其中扮演的角色是**增强者（Augmenter）**，而非替代者。

**一句话概括：** SecMind 让 AI 替你思考"这到底是什么攻击"，告诉你"我为什么这么判断"，但最终的决策权始终在人。

## 二、核心抽象：Alert ≠ Incident ≠ Investigation

这是 SecMind 产品架构的基石。三者边界不清，整个"假设驱动范式"就会退化为"高级 SIEM + AI summarizer"。

### 2.1 Alert（原始告警层）

> 安全设备吐出来的原始信号

- **来源**：Firewall / VPN / Email / EDR / AD / WAF / DLP / NAC
- **特点**：
  - 高噪声、低信噪比
  - 单点事件，缺乏上下文
  - 不可直接决策
- **系统职责**：接收 → 缓存 → 初步结构化 → AI 初筛标签

👉 Alert 层只做一件事：**把原始信号变成可操作的工作项**

### 2.2 Incident（事件聚合层）

> 多个 Alert 聚合后的"安全事件簇"

- **生成方式**：由 Correlation Engine 自动生成，或由分析师手动聚合
- **聚合依据**：
  - 时间窗口（10min / 1h / 24h）
  - 实体关联（同一 IP / 同一用户 / 同一资产）
  - 行为序列（登录失败 → 登录成功 → 横向移动）
  - 攻击模式匹配（rule + embedding similarity）
- **核心属性**：
  - 关联 Alert 列表
  - 涉及实体（User / IP / Device）
  - 攻击可能性评分（attack_likelihood_score）

👉 Incident = **"可能的一次攻击活动"**

### 2.3 Investigation（调查层 / 假设层）

> SecMind 的核心产品层

- 一个 Incident 可生成多个 Hypothesis
- Hypothesis 才是 AI 推理的对象
- 每条 Hypothesis 有独立的置信度、证据链、推理步骤

👉 Investigation = **"对一个事件簇的解释竞争"**

### 三层递进关系

```
Alert（不可决策）→ Incident（结构化事件）→ Hypothesis（多解释竞争）→ Decision（人或 SOAR）
```

**关键设计原则：AI 不能直接从 Alert 跳到 Decision。** 每一层都是不可省略的推理阶梯，确保判断可追溯、可审计、可解释。

## 三、双轨研判机制（Hybrid Intelligence Loop）

告警到达后，到底是自动研判还是人工触发？答案必须是**双轨并行**。

### 3.1 自动研判（System-triggered）

**触发条件（满足任一即触发）：**

| 条件 | 说明 |
|------|------|
| 高风险告警 | riskLevel = Critical / High |
| 多源关联 | ≥3 条 Alert 在同一时间窗口内关联 |
| 行为链完整 | 检测到 login → lateral movement 等完整攻击序列 |
| 攻击模式命中 | 已匹配已知攻击模式（rule + embedding） |

**系统自动执行：**

```
Alert → Correlation Engine 聚合 → Incident 自动生成 → Hypothesis Engine 生成 1~3 个假设
```

自动研判的结果直接进入 `/incidents` 和 `/dashboard`，状态标记为"系统发起"，等待分析师确认。

### 3.2 人工研判（Analyst-triggered）

**触发方式：**

| 入口 | 操作 |
|------|------|
| Alert Inbox | 点击"Investigate"按钮 |
| Alert Inbox | 手动选择多条 Alert → "Create Incident" |
| Alert Stream | 标记升级（Escalate） |
| 任意 Alert 详情 | 点击"启动调查" |

**人工研判流程：**

```
Alert → Inbox 人工筛选 → Incident（未分析）→ Analyst 触发 Hypothesis Engine
```

### 3.3 双轨交汇点

无论自动还是人工，所有 Investigation 最终都汇聚到同一个调查工作台 `/incidents/[id]`。区别仅在于：

| 维度 | 自动研判 | 人工研判 |
|------|----------|----------|
| 发起者 | 系统 | 分析师 |
| 初始状态 | 待确认（预填充假设） | 待分析（空白画布） |
| 分析师角色 | 审核者 — 确认/修正 AI 判断 | 调查者 — 主动构建推理链 |
| 响应速度 | 秒级 | 分钟级 |
| 适用场景 | 高危告警、已知攻击模式 | 低信噪比、新型威胁、分析师直觉 |

## 四、主工作流闭环

这是 SecMind 的核心链路，每一步都有明确的输入输出和责任主体：

```
[Security Devices]
        ↓ 原始日志
     Alerts（/alert-stream 信号流）
        ↓ 接收 + 缓存 + AI初筛
     Alert Inbox（/alert-inbox 运营入口）
        ↓ AI自动聚类 + 人工归类
     Incident 生成
        ↓ Hypothesis Engine（多假设生成）
     Investigation（/incidents/[id] 调查工作台）
        ↓ Analyst 研判 + Copilot 解释
     Decision（/copilot 可解释性中心）
        ↓ SOAR 执行
     Response（/response 自动响应）
        ↓ 反馈回流
     Evolution Engine（假设置信度演化）
```

**闭环关键：** 响应执行结果反馈至 Evolution Engine，动态调整假设置信度。如果"冻结账号"后异常行为消失，H-A（账号入侵）置信度上升；如果异常持续，则可能触发新假设。

## 六、身份认证

安全运营的第一道门，兼顾安全性与便捷性：

- **标准登录**：用户名/邮箱 + 密码，支持"记住我"

## 七、核心页面详解

### 7.1 告警收件箱 · `/alert-inbox` 🆕

> 安全运营收件箱（SOC Inbox）—— 告警太多，从这里开始。

这是 SecMind 的**运营入口级页面**。不是日志流，而是**待处理任务队列**——解决"告警太多，用户不知道从哪里开始"的核心痛点。

#### 左侧 — 告警队列（核心）

卡片式列表，每张告警卡片包含：

| 字段 | 说明 |
|------|------|
| Alert ID | 唯一标识 |
| 来源 | EDR / VPN / Email / Firewall / AD / WAF |
| 触发时间 | 精确到秒 |
| 风险等级 | Critical / High / Medium / Low / Info |
| AI 初筛标签 | ⚠️ 可疑 / 🔴 高危 / 🟡 待确认 / 🟢 噪声 |
| 关联实体 | User / IP / Device |
| 聚类归属 | 已归类的 Incident ID（如有） |

**筛选与排序：**

| 维度 | 选项 |
|------|------|
| AI 初筛 | 全部 / 高危 / 可疑 / 待确认 / 噪声 |
| 来源 | 全部 / EDR / VPN / Email / Firewall / AD |
| 风险等级 | Critical / High / Medium / Low |
| 归类状态 | 未归类 / 已归类 |
| 排序 | 时间倒序 / 风险优先 / AI评分优先 |

#### 右侧 — 处理动作面板

选中任意 Alert 后，右侧展示操作面板：

**动作 1 — 加入/生成 Incident（核心动作）**

- **Add to existing Incident**：从已有 Incident 列表中选择，将此 Alert 归入
- **Create new Incident**：以此 Alert 为种子，创建新 Incident，系统自动关联同实体/同时间窗口的其他 Alert

**动作 2 — 快速研判**

- **Ask AI**：
  - "Is this part of an attack?" — AI 判断此 Alert 是否属于已知攻击模式
  - "Correlate with existing incidents" — AI 搜索已有 Incident，判断是否应归入
- AI 返回结果以建议形式呈现，**不直接替代决策**

**动作 3 — 标记处理状态**

| 标记 | 含义 | 后续动作 |
|------|------|----------|
| 🔴 Escalate | 升级处理 | 自动触发 Incident 创建 |
| ⚠️ Suspicious | 标记可疑 | 留在队列，等待更多关联 |
| 🟢 Noise | 标记噪声 | 移出主队列，进入噪声池 |
| ✅ False Positive | 误报确认 | 移出队列，反馈至 AI 模型 |

**动作 4 — 启动调查**

- 点击 **Investigate** 按钮，直接从此 Alert 进入调查流程
- 系统自动创建 Incident + 触发 Hypothesis Engine

#### 为什么需要 Inbox？

| 没有 Inbox | 有 Inbox |
|------------|----------|
| 告警太多，不知道从哪开始 | AI 初筛排序，高危优先处理 |
| Alert 和 Incident 混在一起 | 三层抽象清晰，Alert 是 Alert，Incident 是 Incident |
| AI 直接给结论，人无法介入 | AI 辅助归类，人做最终决策 |
| 缺少"收件箱"的任务感 | 每条 Alert 都是待办，处理完即消失 |

### 7.2 信号流 · `/alert-stream`

> 原始信号，实时涌流。

这是安全数据的"第一现场"——来自防火墙、VPN、邮件网关、EDR、AD 等安全设备的原始告警信号，以实时流的方式持续推送。定位为**技术层监控视图**，与运营层的 Inbox 互补。

**实时体验**

- 每5秒自动推入新信号，新条目带青色渐入高亮动画
- 顶部 **LIVE** 脉冲指示灯，确认数据链路在线
- 滑动窗口保留最近50条，自动淘汰过期数据

**多维度过滤**

| 维度 | 选项 |
|------|------|
| 来源 | 防火墙 🔴 / VPN 🟡 / 邮件 🔵 / EDR 🟣 / AD 🟠 |
| 级别 | 严重（红色发光）/ 警告（黄色发光）/ 信息（青色半透明） |
| 关键词 | 自由文本搜索，匹配日志内容 |

每条信号以紧凑的单行格式呈现：时间戳 · 来源标签 · 级别指示灯 · 原始日志，兼顾信息密度与可读性。

**从信号流到 Inbox：** 任意信号可点击"加入 Inbox"或"升级处理"，将原始信号转化为可操作的工作项。

### 7.3 告警详情 · `/alerts/[id]`

> 深度剖析，一告警一世界。

点击任意告警，进入深度分析视图。三栏布局，左中右各司其职：

#### 左栏 · 时间线与原始数据

- **事件时间线**：纵向时间轴，每个节点按类型着色
  - 🔴 告警触发 — 攻击信号
  - ⚪ 日志记录 — 系统行为
  - 🟡 响应操作 — 人工/AI动作
  - 🔵 AI分析 — 智能研判

  节点间以渐变连接线串联，还原事件完整脉络。

- **原始日志**：终端风格（黑底绿字）展示原始数据，一键复制，方便留存取证。

#### 中栏 · AI 研判与攻击链（核心）

**事件摘要**：告警全貌一目了然——标题、类型图标、风险等级（高危以上脉冲闪烁）、状态、描述、IP流向（源 → 目标）、时间戳、标签。

**AI 研判分析**（平台最核心的能力展示）：

- **风险评分仪表盘**：SVG 圆环图，0-100分动态渲染。高分区域带脉冲动画，视觉上传递紧迫感
- **AI 结论**：用自然语言阐述"发生了什么"
- **攻击链推理**：将散点事件串联为攻击故事。如：`钓鱼邮件 → 凭证窃取 → VPN接入 → C2通信`，起始节点红色高亮，末端粉色标识
- **AI 处置建议**：按优先级排列（紧急/高/中/低），每条建议带编号和优先级标签，可直接指导行动
- **AI Agent 标识**：标注分析所用的模型（如 GPT-4o）

**关联事件**：与此告警存在关联的其他安全事件，帮助发现攻击全貌。

**Incident 归属信息**：如果此 Alert 已被归入某个 Incident，显示 Incident ID + 关联假设 + 置信度，支持一键跳转至调查工作台。

#### 右栏 · 用户画像与工单

**用户画像**——将安全事件与"人"关联：

- 基本信息：头像、姓名、邮箱
- 风险标签：🔴 敏感人员 / 🟡 休假中 / ⚫ 已离职
- **智能异常提示**：当检测到"休假中却存在登录行为"或"登录地与办公地不一致"时，自动以黄色警告框突出显示
- 详细属性：部门、岗位、职级、直属领导、办公地点、最近登录地、休假/在职状态

**历史事件**：该用户过往的安全事件记录，辅助判断是否为惯犯或长期目标。

**ITSM 工单**：关联的处置工单，支持一键创建新工单，打通安全运营与IT服务管理。

### 7.4 安全调查入口 · `/dashboard`

> 你的安全指挥中心，一眼掌握全局态势。

这是分析师每天工作的起点。页面以"调查任务"为核心组织信息，而非传统的告警列表——因为真正重要的是威胁本身，而非单条告警。

**态势感知区**

四张统计卡片，快速感知当前安全态势：

| 指标 | 含义 | 颜色语义 |
|------|------|----------|
| 正在调查 | 当前活跃的调查任务数 | 青色 — 进行中 |
| 待研判攻击 | 等待分析师确认的攻击假设 | 琥珀色 — 待决策 |
| 已收集证据链 | 已归集的攻击证据总数 | 绿色 — 进展顺利 |
| 已执行响应 | 已完成的自动化响应操作 | 红色 — 已行动 |

**活跃调查列表**

每张调查卡片是一个"威胁故事"的缩影：

- **调查编号**：唯一标识（如 INV-2026-0042）
- **发起方式**：🤖 系统自动 / 👤 分析师发起
- **攻击链路径**：可视化展示攻击者的行动轨迹（VPN → Email → PowerShell → C2），一眼看清攻击全貌
- **主攻击研判**：AI 给出的最可能攻击类型，附带置信度进度条
  - ≥80% 红色 — 高度确信，需立即行动
  - ≥60% 琥珀色 — 中等确信，需持续关注
  - <60% 青色 — 初步判断，需更多证据
- **状态徽章**：调查中 / 分析中 / 已升级
- **证据概览**：事件数、关联对象数、最后更新时间
- **一键继续**：跳转至调查工作台深入分析

**信息侧栏**

- **最近活动**：实时流式展示假设更新、证据归集、响应执行等动态，保持团队信息同步
- **升级告警**：红色高亮展示已升级的紧急事件，确保不遗漏任何高危威胁
- **快捷链接**：快速跳转至告警收件箱、调查中心、研判实验室、证据看板、响应操作

### 7.5 安全调查中心 · `/incidents`

> 所有调查任务，尽在掌控。

这是调查任务的统一管理入口，提供强大的筛选和检索能力。

**多维筛选体系**

### 7.7 AI 研判助手 · `/copilot`

> 打开 AI 的"思维黑盒"，让每一次判断都可解释、可审计。

这是 SecMind 的**AI 可解释性中心**。安全运营最怕"黑盒判断"——AI 说这是账号入侵，但为什么？Copilot 页面完整呈现 AI 的推理过程。

**左侧（60%）— 证据分析**

**底部 — 证据矩阵**

## 八、核心引擎体系

> SecMind 的"大脑"，让安全分析从经验驱动走向推理驱动。

引擎体系由7个模块协同工作，覆盖从原始信号到响应决策的完整链路：

### 引擎协作流程

```
[Security Devices]
        ↓ 原始日志
   Adapters（适配器）→ 统一 SecurityEvent 格式
        ↓
   Correlation Engine（聚类引擎）🆕 → Incident 生成
        ↓
   Hypothesis Engine（假设引擎）→ 多假设生成
        ↓
   Scorer（评分器）→ 置信度计算
        ↓
   Evolution Engine（进化引擎）→ 置信度持续演化
        ↓
   Graph Store（图谱存储）→ 关系网络构建
        ↓
   反馈回流 → Evolution Engine 更新假设
```

### 模块详解

**适配器 · `adapters.ts`**

数据统一的桥梁。将告警(Alert)、VPN会话、邮件日志、登录记录等异构数据源转换为统一的 SecurityEvent 格式，让推理引擎无需关心数据来源差异。

**聚类引擎 · `correlation-engine.ts`** 🆕

Alert → Incident 的关键跃迁引擎。负责将散点告警聚合为结构化事件簇：

- **时间窗口聚合**：同一时间窗口（10min/1h/24h）内的告警自动分组
- **实体图谱聚合**：共享同一 IP / User / Device 的告警关联
- **攻击模式匹配**：基于预定义规则 + embedding similarity 匹配已知攻击模式
- **行为序列检测**：识别 login → lateral movement 等多步攻击序列

输出结构：

```
Incident {
  id: string
  alerts: Alert[]              // 聚合的原始告警
  entities: Entity[]           // 涉及的实体（User/IP/Device）
  attack_likelihood_score: number  // 攻击可能性评分
  correlation_method: string   // 聚合方式（auto/manual）
  created_by: 'system' | 'analyst'
  status: 'pending' | 'investigating' | 'resolved' | 'false_positive'
}
```

**假设引擎 · `engine.ts`**

推理的起点。基于预定义规则对 Incident 中的安全事件进行模式匹配，自动生成证据和推理步骤，构建攻击路径，输出多个竞争假设。核心机制是**多假设竞争**——不预设结论，让证据决定方向。

**进化引擎 · `evolution-engine.ts`**

推理的持续动力。每当新事件注入或响应反馈回流，进化引擎动态更新各假设置信度，评估新证据的影响，重新排序假设，并生成演化解释——"为什么置信度从78%上升到92%？"每一步变化都有据可查。

**评分器 · `scorer.ts`**

置信度的计算核心。综合考量三个维度：
- **时间衰减**：越久远的证据权重越低
- **证据相关性**：与假设直接相关的证据权重更高
- **反对惩罚**：矛盾证据显著降低置信度

最终输出可解释的评分和排序。

**图谱存储 · `graph-store.ts`**

安全关系网络。以图结构存储实体（用户、主机、IP）和关系（连接、访问、认证），支持路径查找算法，用于发现攻击链中的隐藏关联。

**类型定义 · `types.ts`**

整个推理系统的数据契约——Evidence（证据）、Hypothesis（假设）、SecurityEvent（安全事件）、Incident（事件簇）、AttackPath（攻击路径）、InferenceStep（推理步骤），确保模块间类型安全。

## 十二、设计哲学

### 八大核心创新

1. **三层抽象，边界清晰**
   Alert ≠ Incident ≠ Investigation。每一层有明确的输入输出和责任边界，AI 不直接从原始信号跳到结论，确保推理链完整可审计。

2. **双轨研判，人机协同**
   自动研判处理高危和已知模式，人工研判处理低信噪比和新型威胁。AI 是增强者，不是替代者——系统自动生成假设，分析师做最终决策。

3. **假设驱动，而非告警驱动**
   传统 SOC 以告警为起点，分析师被动响应。SecMind 以攻击假设为核心，AI 主动建模、主动推理、主动建议，分析师从"告警消防员"变为"威胁猎手"。

4. **证据链可追溯**
   每条证据明确标注支持/反对哪个假设及权重，推理链完整可审计。不再有"AI 说是就是"的黑盒，每一个判断都有据可查。

5. **AI 可解释性**
   Copilot 页面完整呈现 AI 的推理过程——证据贡献度、假设对比矩阵、MITRE ATT&CK 映射。让分析师理解"为什么"，而不仅仅是"是什么"。

6. **假设-响应联动**
   SOAR 操作与假设置信度深度绑定。高置信度假设触发关键响应，低置信度假设仅建议监控。响应有据，行动有度。响应结果反馈至进化引擎，形成闭环。

7. **用户画像增强**
   融合 HR 数据（休假状态、离职标记、办公地点），让安全判断超越技术指标，纳入"人"的维度——"休假中的账号被登录"比"异常IP登录"更有威胁意义。

8. **知识库驱动 AI**
   RAG 知识库为 AI 研判注入领域知识——威胁情报、应急SOP、安全基线、合规要求、漏洞库、处置手册。AI 不再是通用模型，而是懂安全的专家。
