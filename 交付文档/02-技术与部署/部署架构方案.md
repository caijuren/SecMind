### 2.1 单机部署 (All-in-One)

#### 适用场景
- 概念验证（POC）环境
- 开发/测试环境
- 小微企业用户量低于50人的场景
- 客户评测阶段

#### 最低配置要求
| 组件 | 配置要求 |
|------|---------|
| CPU | 8核 (x86_64 / ARM64) |
| 内存 | 32 GB |
| 磁盘 | 500 GB SSD |
| 带宽 | 50 Mbps |

### 2.2 高可用部署 (High Availability)

#### 适用场景
- 生产环境
- 月活跃用户 500 - 5000
- SLA要求 99.9% 以上
- 企业级安全运营中心（SOC）

#### 架构说明

```
                          +---------+
                          |   DNS    |
                          | 轮询/GSLB|
                          +----+----+
                               |
                    +----------+----------+
                    |      Nginx 集群      |
                    | (2台, Keepalived    |
                    | 主备 + 健康检查)     |
                    +----------+----------+
                               |
         +---------------------+---------------------+
         |                     |                     |
   +-----v-----+        +-----v-----+        +-----v-----+
   | FastAPI   |        | FastAPI   |        | FastAPI   |
   | App Pod 1 |<------>| App Pod 2 |<------>| App Pod N |
   | (4副本)   |        | (4副本)   |        | (4副本)   |
   +-----------+        +-----------+        +-----------+
         |                     |                     |
         +---------------------+---------------------+
                               |
         +---------------------+---------------------+
         |                                           |
   +-----v---------------------+     +---------------v------+
   | PostgreSQL 主从复制        |     |  Redis 哨兵模式      |
   |  +-- Primary (读写)       |     |  +-- Master (读写)   |
   |  +-- Standby-1 (只读)    |     |  +-- Replica-1       |
   |  +-- Standby-2 (只读)    |     |  +-- Replica-2       |
   |  +-- Standby-3 (灾备)    |     |  +-- Sentinel x3     |
   +---------------------------+     +----------------------+
         |
   +-----v---------------------+
   |  对象存储集群 MinIO        |
   |  (4节点, Erasure Coding)  |
   +---------------------------+
```

#### 节点配置建议

| 节点角色 | 服务器数量 | CPU | 内存 | 磁盘 | 说明 |
|---------|-----------|-----|------|------|------|
| Nginx LB | 2 | 4核 | 8 GB | 100 GB | Keepalived主备 |
| API Server | 4+ | 16核 | 64 GB | 200 GB SSD | Docker Swarm/K8s集群 |
| PostgreSQL | 4 | 16核 | 64 GB | 2 TB SSD | 1主3从 |
| Redis | 3 | 8核 | 32 GB | 100 GB SSD+NVMe | 1主2从+3Sentinel |
| MinIO | 4 | 8核 | 32 GB | 4 TB HDD+NVMe | Erasure Coding(4+2) |
| 监控节点 | 1 | 8核 | 16 GB | 500 GB | Prometheus+Grafana |
| 合计 | 18+ | - | - | - | 总服务器约18-20台 |

#### 部署要点
```bash
# 1. 配置负载均衡 (Nginx + Keepalived)
# 在主备Nginx节点配置
cat > /etc/nginx/conf.d/secmind.conf << 'EOF'
upstream secmind_api {
    least_conn;
    server 10.0.1.10:8000 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:8000 max_fails=3 fail_timeout=30s;
    server 10.0.1.12:8000 max_fails=3 fail_timeout=30s;
    server 10.0.1.13:8000 max_fails=3 fail_timeout=30s;
    keepalive 32;
}

server {
    listen 443 ssl http2;
    server_name secmind.company.com;

    location /api/ {
        proxy_pass http://secmind_api;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 120s;
        proxy_connect_timeout 10s;
    }

    location /ws/ {
        proxy_pass http://secmind_api;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 3600s;
    }
}
EOF

# 2. 配置 PostgreSQL 主从同步
# 主库 postgresql.conf
wal_level = replica
max_wal_senders = 6
wal_keep_size = 1024   # 保留1GB WAL

# 3. 从库恢复配置
pg_basebackup -h primary_ip -D /var/lib/postgresql/data -U replicator -P -v
```

#### SLA 达成策略
| 故障场景 | 切换时间 | 数据丢失容忍 |
|---------|---------|-------------|
| API Pod 故障 | < 10秒 | 无数据丢失 |
| Nginx 主故障 | < 5秒 | 无影响 |
| PostgreSQL 主故障 | < 30秒 | < 1MB (异步复制) |
| Redis 主故障 | < 10秒 | 取决于持久化配置 |
| 整机房故障 | < 5分钟 | < 1分钟数据 |

### 2.3 分布式多Region部署 (Multi-Region)

#### 适用场景
- 跨国企业/全球化部署
- 月活跃用户 10000+
- 数据主权合规要求（GDPR、等保、PIPL）
- SLA要求 99.99% 以上

#### 架构说明

```
                              +-----------------------+
                              |  Global DNS / 智能调度  |
                              |  AWS Route53 / 阿里云DNS |
                              +----------+------------+
                                         |
            +----------------------------+----------------------------+
            |                            |                            |
   +--------v--------+          +--------v--------+          +--------v--------+
   |   Region 1      |          |   Region 2      |          |   Region 3      |
   |  (华北-北京)     |          |  (华东-上海)     |          |  (华南-广州)     |
   |                 |          |                 |          |                 |
   | +-------------+ |          | +-------------+ |          | +-------------+ |
   | |  WAF/LB     | |          | |  WAF/LB     | |          | |  WAF/LB     | |
   | +------+------+ |          | +------+------+ |          | +------+------+ |
   |        |        |          |        |        |          |        |        |
   | +------v------+ |          | +------v------+ |          | +------v------+ |
   | | API集群 x N | |          | | API集群 x N | |          | | API集群 x N | |
   | +------+------+ |          | +------+------+ |          | +------+------+ |
   |        |        |          |        |        |          |        |        |
   | +------v------+ |          | +------v------+ |          | +------v------+ |
   | | PostgreSQL  | |          | | PostgreSQL  | |          | | PostgreSQL  | |
   | | 本地主从     | |          | | 本地主从     | |          | | 本地主从     | |
   | +------+------+ |          | +------+------+ |          | +------+------+ |
   +--------+--------+          +--------+--------+          +--------+--------+
            |                            |                            |
            +----------------------------+----------------------------+
                                         |
                           +-------------v-------------+
                           | 跨区域数据同步层            |
                           | PostgreSQL 逻辑复制         |
                           | 或 Kafka 跨Region数据桥接     |
                           +-------------+-------------+
                                         |
                           +-------------v-------------+
                           |    全局管控中心 (Global)    |
                           |  - 多租户元数据同步              |
                           |  - 全局威胁情报聚合         |
                           |  - 跨Region报表             |
                           +---------------------------+
```

#### 全球加速方案
| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 全球DNS | AWS Route53 / 阿里云DNS | 基于延迟/Latency的智能解析 |
| CDN | Cloudflare / 阿里云CDN | 静态资源全球加速 |
| 动态加速 | 专用动态加速线路 | API动态请求加速 |
| 跨区域互联 | 云厂商专线 / SD-WAN | Region间低延迟互联 |

#### 数据同步策略
| 数据类别 | 同步方式 | 同步延迟 | 说明 |
|---------|---------|---------|------|
| 用户/租户数据 | 主Region写入, 逻辑复制 | < 1秒 | 仅主Region可写 |
| 告警事件 | 本地写入, 异步汇总 | < 5秒 | 本地优先 |
| 威胁情报 | 全局发布, 本地订阅 | < 10秒 | Kafka广播 |
| AI分析结果 | 本地缓存, 异步同步 | < 30秒 | 最终一致性 |
| 配置文件 | 全局配置中心 | 实时 | etcd/Nacos |

#### 数据主权合规
```yaml
数据分类存储策略:
  用户身份数据:
    存储位置: 用户所在Region
    出境限制: 禁止出境
  安全告警数据:
    存储位置: 本地Region
    出境限制: 脱敏后可汇总至Global
  AI推理日志:
    存储位置: 本地Region
    出境限制: 保留90天后可删除
  操作审计日志:
    存储位置: 本地Region
    出境限制: 不可出境
```

## 5. 数据库架构

### 5.1 PostgreSQL 主从复制架构

```
+------------------+                +------------------+
|  Primary (读写)   |                |  Standby-1 (只读) |
|  10.0.0.1.10:5432   |<---------------| 10.0.1.11:5432   |
|  - 所有写入操作  |  Streaming     |  - 报表查询       |
|  - DDL/DML      |  异步/同步复制  |  - 历史数据查询    |
|  - 事务处理      |                |  - AI分析只读查询  |
+------------------+                +------------------+
        |                                    |
        |  Streaming Replication              |  Cascade Replication
        |                                    |
+------------------+                +------------------+
|  Standby-2 (只读) |                |  Standby-3 (灾备) |
| 10.0.1.12:5432   |                | 10.0.2.10:5432   |
|  - 实时同步       |                |  - 异地灾备       |
|  - 可提升为主库   |                |  - 延迟同步       |
+------------------+                +------------------+
```

### 5.2 备份策略

| 备份类型 | 工具 | 频率 | 保留周期 | 存储位置 |
|---------|------|------|---------|---------|
| 全量备份 | pg_dump / pg_basebackup | 每日凌晨2:00 | 30天 | MinIO对象存储 |
| WAL归档 | pg_archive | 连续归档 | 连续 | 7天 | MinIO对象存储 |
| 增量备份 | WAL归档基础上 | 每30分钟 | 7天 | MinIO对象存储 |
| 逻辑备份 | pg_dump (按Schema) | 每周日凌晨4:00 | 90天 | 异地存储 |
| 表空间快照 | LVM快照 | 每日 | 3天 | 本地磁盘 |

### 5.3 备份命令示例
```bash
# 全量备份 (pg_basebackup)
pg_basebackup -h 10.0.1.10 -D /backup/secmind_full_$(date +%Y%m%d) \
  -U backup_user -P -v --wal-method=stream

# 逻辑备份 (特定Schema)
pg_dump -h 10.0.1.10 -U backup_user -d secmind -n public \
  -F c -f /backup/secmind_public_$(date +%Y%m%d).dump

# WAL归档配置 (postgresql.conf)
archive_mode = on
archive_command = 'cp %p /archive/%f && mc cp /archive/%f minio/secmind-backups/wal/%f'
archive_timeout = 60  # 每60秒归档
```

### 5.4 恢复流程
```bash
# 时间点恢复 (PITR)
# 1. 停止目标数据库
systemctl stop postgresql

# 2. 恢复全量备份
tar -xzf /backup/secmind_full_20250101/base.tar.gz -C /var/lib/postgresql/15/data/

# 3. 配置恢复目标时间
cat > /var/lib/postgresql15/data/recovery.conf << 'EOF'
restore_command = 'mc cp minio/secmind-backups/wal-backups/wal/%f %p'
recovery_target_time = '2025-01-01 14:30:00 CST'
recovery_target_action = 'promote'
EOF

# 4. 启动数据库进入恢复模式
systemctl start postgresql

# 5. 验证恢复结果
psql -c "SELECT now();"
```

## 9. 扩容方案

### 9.1 水平扩容 (Scale-out)

#### API 服务水平扩容
```
扩容前: [Pod] -> [Pod] [Pod] [Pod]          (3副本)
扩容后: [Pod] [Pod] [Pod] [Pod] [Pod] (5副本)

操作步骤:
1. docker compose up -d --scale api-server=5 --no-deps
   或 Kubernetes: kubectl scale deployment api-server --replicas=5
2. Nginx/LB自动发现新Pod (健康检查)
3. 观察监控指标确认负载均匀
4. 建议每次增减不超过50%副本数
```

#### PostgreSQL 只读从库扩容
```bash
# 1. 添加新的从库节点
pg_basebackup -h primary_host -D /var/lib/postgresql/data \
  -U replicator -P -v -X stream

# 2. 配置 postgresql.conf
primary_conninfo = 'host=primary_host port=5432 user=replicator password=xxx'
primary_slot_name = 'replica_standby_4'

# 3. 启动从库
pg_ctl start -D /var/lib/postgresql/data

#4. 在应用层配置连接池，增加从库连接
```

#### Redis 集群扩容
```bash
# Redis Cluster 模式节点添加
redis-cli --cluster add-node NEW_NODE_IP:6379 EXISTING_NODE_IP:6379

# 重新分片
redis-cli --cluster reshard EXISTING_NODE_IP:6379
```

### 9.2 垂直扩容 (Scale-up)

| 组件 | 扩容内容 | 影响 | 操作方式 |
|------|---------|------|---------|
| API 服务器 | CPU/内存升级 | 需重启服务 | 滚动更新,零停机 |
| PostgreSQL | CPU/内存/磁盘 | 需重启数据库 | 主从切换后,主备切换 |
| Redis | 内存增加 | 无需重启(热扩容) | `config set maxmemory` |
| MinIO | 磁盘扩容 | 在线扩容 | 新增磁盘,自动均衡 |

## 附录：灾备与业务连续性方案（强化版）

### 灾备等级定义
| 等级 | RPO（恢复点目标） | RTO（恢复时间目标） | 适用场景 |
|------|------------------|-------------------|---------|
| L1 基础级 | 24小时 | 48小时 | 非关键业务 |
| L2 标准级 | 4小时 | 8小时 | 一般生产环境 |
| L3 企业级 | 1小时 | 30分钟 | 核心生产环境 |
| L4 金融级 | 15分钟 | 5分钟 | 关键业务系统 |

### 同城双活方案
适用场景：金融、政务等对高可用要求严格的客户。
- 在同城两个数据中心各部署一套完整系统
- 数据库采用主主复制或流复制
- Redis采用集群模式跨数据中心部署
- 前端通过全局负载均衡（GSLB）分发流量
- 任意数据中心故障时，流量自动切换至另一中心
- 切换时间：<30秒

### 异地灾备方案
适用场景：有异地容灾合规要求的客户。
- 主中心（同城）承载生产流量
- 备中心（异地，建议距离>300km）承载备份
- 数据库通过流复制异步同步至备中心
- 备中心保持热备状态，不承载流量
- 主中心整体故障时，手动或自动切换至异地灾备中心
- 切换时间：<15分钟

### 数据备份策略
| 备份类型 | 周期 | 保留策略 | 存储位置 |
|---------|------|---------|---------|
| 全量备份 | 每日凌晨 | 保留30天 | 本地+异地 |
| 增量备份 | 每4小时 | 保留7天 | 本地 |
| WAL归档 | 连续 | 保留7天 | 本地+异地 |
| 配置文件备份 | 每次变更后 | 保留10个版本 | 代码仓库 |
| 容器镜像备份 | 每次发布 | 保留最近5个版本 | 镜像仓库 |

### 灾难恢复演练
- 演练频率：每季度一次桌面推演，每年一次实战演练
- 演练范围：数据库故障切换、Redis集群故障切换、应用服务器故障切换、全站点切换
- 演练记录及改进措施

### 故障切换决策树
检测到服务不可用
├── 单点故障 → 重启服务（5分钟内3次）
│   ├── 恢复成功 → 结束
│   └── 恢复失败 → 启动备用实例
├── 模块级故障 → 切换至备用模块
├── 数据中心级故障 → 启动灾备切换
└── 其他 → 逐步排查基础设施
