## 1. 测试概述

### 1.1 测试目的

本次性能基准测试旨在全面评估 SecMind 平台在信创硬件环境下的各项性能表现，达成以下目标：

- **定量评估**：获取平台在典型信创环境下的核心性能指标，包括告警处理能力、API 吞吐量、并发用户支持能力、AI 推理延迟等关键数据；
- **资源规划参考**：为不同规模客户提供准确的硬件配置建议与容量规划依据；
- **信创兼容性验证**：对比信创环境（鲲鹏 + openEuler + KingbaseES）与 x86 环境（Intel + CentOS + PostgreSQL）的性能差异，验证信创环境的生产就绪性；
- **优化方向确定**：识别性能瓶颈，为后续性能优化工作提供数据支撑与方向指引。

### 1.2 适用范围

本报告适用于以下场景：

- 面向大型客户的售前方案设计，提供容量规划与性能承诺依据；
- 信创环境部署前的性能评估与资源规划；
- 平台版本迭代过程中的性能回归验证基线。

### 1.3 测试周期

| 阶段 | 时间 | 说明 |
|------|------|------|
| 测试准备 | 2026-05-10 ~ 2026-05-12 | 环境搭建、数据准备、脚本开发 |
| 基准测试执行 | 2026-05-13 ~ 2026-05-16 | 各场景测试执行与数据采集 |
| 数据分析与报告编制报告 | 2026-05-17 ~ 2026-05-18 | 数据整理、分析、报告编制 |

## 3. 测试方法与工具

### 3.1 测试工具

- **Apache JMeter 5.6.3**：HTTP API 压测、WebSocket 连接模拟、数据库压测脚本管理；
-  **Prometheus + Grafana**：服务器资源监控 (CPU / 内存 / 磁盘 IO / 网络)
- **Arthas**：Java 应用在线诊断、线程栈分析
- **perf / vnstat / iostat**：Linux 内核级性能分析
- **Redis-benchmark**：缓存层性能验证
- **自定义数据构造器**：基于 Go 编写的告警流模拟器，支持指定 EPS、告警类型分布

### 6.4 差异分析

#### 性能差异的主要来源

1. **指令架构差异**：ARM64 与 x86 之间约 5-8% 的指令执行效率差异属于正常硬件代差，在计算密集型场景（如压缩/加密）中差距扩大；
2. **数据库兼容层开销**：KingbaseES 在兼容 PostgreSQL 协议时增加了语法解析与安全审计层，约有 47% 的额外开销；
3. **JDK 差异**：毕昇JDK 基于 OpenJDK，在 ARM64 上的即时编译（JIT）优化深度略低于 Oracle JDK on x86，差距约在 3-5%；
4. **内核调度差异**：openEuler 的 CPU 调度策略与 CentOS 存在微小差异，在 在高并发场景下体现为上下文切换开销增大。

#### 结论

**整体性能差异在 5~10%，在可接受范围内。** 信创环境完全可以承载大型客户的生产负载。对于主要瓶颈（数据库查询、高并发场景），可以通过优化配置和应用层手段弥合差距。

## 7. 性能优化建议

### 7.1 数据库连接池配置建议

#### 当前配置问题

- 默认连接池大小为 20，在 500 并发用户下频繁等待连接；
- 长，复杂查询等待造成连接池阻塞。

#### 推荐配置

参数 | 推荐值 | 说明 |
|------|---------|------|
| `maximumPoolSize` | 50 ~ 100 | 根据节点内存调整，每连接约 10MB |
| `minimumIdle` | 10 | 保持最小空闲保留，避免频繁创建 |
| `connectionTimeout` | 5000ms | 超时缩短，快速失败 |
| `idleTimeout` | 600000ms | 10 分钟回收空闲连接 |
| `maxLifetime` | 1800000ms | 30 分钟强制重建，避免连接泄漏 |
| `leakDetectionThreshold` | 10000ms | 开启连接泄漏检测 |

#### SQL 优化建议

| 发现的问题 | 优化建议 | 预期提升 |
|-----------|-------------------|--------|
| 告警搜索未使用覆盖索引 | 新增复合索引 `(created_at, severity, status)` | P99 降低 40% |
| 聚合查询使用临时文 | 调整 `work_mem` 至 64MB，启用并行聚合 | 延迟降低 60% |
| 模糊搜索未走索引 | 使用 pg_trgm 扩展或 KingbaseES 全文索引 | 延迟降低 90% |
| 分页查询 OFFSET 过大 | 改用 Keyset Pagination → Keyset Pagination（游标分页） | 深层分页性能提升 100 倍 |

### 7.2 Redis 缓存策略建议

#### 现状

- 告警统计缓存 TTL 过长（300s），导致数据时效性不足；
- 热点 key 集中在仪表盘维数据，部分 key 单 key 达到 MB 级别；
- 无主动失效机制，强依赖 TTL 淘汰。

#### 推荐优化

| 优化项 | 方案 | 预期收益 |
|--------|------|----------|
| **合理设置 TTL** | 动态 TTL：短周期聚合数据 10s，配置 300s，长周期统计 60s | 增加内存利用率，减少旧数据返回 |
| **热点 Key 拆分 | 将大 key（如全量统计）拆分为多维度小 key | 减少序列化/反序列化开销，降低单次请求延迟 30% |
| **缓存策略** | 写时失效 + 延迟双删 + 异步重建 | 降低缓存击穿概率 95% |
| **热 key 治理** | 采用本地缓存（Caffeine）+ Redis 多级缓存架构 | 热 key 查询延迟 < 1ms |
| **避免淘汰策略导致的性能抖动** | 配置 `maxmemory-policy allkeys-lru`，预留 20% 内存作为安全缓冲区 | 防止淘汰风暴 |

### 7.3 AI 推理批量处理优化

#### 推理延迟瓶颈分析

| 瓶颈点 | 说明 |
|-----------|------|
| **显存带宽限制** | 昇腾 910B 单卡显存带宽 ~1.2 TB/s，在长序列场景中显存访问成为主要瓶颈 |
| **批量处理开销** | Batch size 增大后，Padding 导致无效计算量上升，线性加速比下降 |
| **推理框架** | vLLM 的 PagedAttention 在 ARM64 上的优化程度不足 |

#### 优化方案

|  | 方案 | 预期效果 |
|--------|--------|----------|
| **动态 Batch** | 启用 vLLM `--max-num-batched-tokens` 动态调整 Batch| 吞吐提升 30% |
| **Continuous Batching** | 开启** | vLLM 默认支持，确保已开启 | 减少碎片等待时间 |
| **模型量化** | INT4 量化（AWQ 或 GPTQ）用于生产环境 | 显占用降低 ~60%，延迟降低 ~30%，精度损失< 1% |
| **前缀缓存** | 对系统提示词（System Prompt）进行 KV Cache 复用 | 固定提示词场景推理速度提升 2 倍 |
| **分离部署** | 推理服务独立节点，与主应用分离 | 避免推理高峰抢占 CPU 资源 |
| **异步批量请求** | 收集 50ms 窗口内推理请求，合并为一个 Batch | Batch size 填充率提升 40% |

### 7.4 告警聚合窗口优化

#### 现状

- 所有聚合规则使用固定窗口（Aggregation Rule）使用统一的窗口长度（5 分钟），导致：
  - 短周期规则（如 1 分钟内同一源 IP 登录失败 5 次）延迟过高；
 长周期规则（如 30 分钟内横向移动检测）窗口过早结束，漏报率高。

#### 推荐

推荐配置

| 规则类型 | 推荐窗口 | 聚合维度 | 说明 |
|---------|---------|---------|------|
| 暴力破解检测 | 1 分钟 | source IP | 短窗口，快速响应 |
| 登录异常 | 5 分钟 | account + source IP |  source IP | 标准窗口 |
| 横向移动检测 | 30 分钟 | source IP + destination IP | 长窗口，全面覆盖 |
| DDoS 检测 | 1 分钟 | 5-tuple | 短窗口，实时检测 |
| 数据外泄检测 | 10 分钟 | destination domain | |

#### 计算资源优化

| 优化项 | 方案 |
|--------|------|
| **窗口状态存储** | 将聚合窗口状态移至 Redis，利用过期机制 (EXPIRE) 自动清理 |
| **增量聚合** | 不再每次全量扫描窗口数据，使用计数器 + 滑动窗口算法 |
| **窗口内存限制** | Redis 单窗口最大 50MB，超出则触发降级（降采样） |

## 8. 测试结论

### 8.1 结论总述

本次性能基准测试在信创环境（鲲鹏 920 + openEuler + KingbaseES）和 x86 对比环境（Intel Xeon + CentOS + PostgreSQL）上对 SecMind 平台进行了全面评估，覆盖告警处理、API 吞吐、并发用户、AI 推理、数据库查询、WebSocket 推送、页面加载等核心场景。

**总体结论：SecMind 平台在信创环境下表现优异，核心指标达到或超过设计目标，具备面向大型客户的生产交付条件。**

### 8.2 关键结论

| 维度 | 结论 |
|------|------|
| **告警处理能力** | 单节点稳定支持系统 5,000 EPS，P99 延迟 < 500ms，满足大型客户需求；超过 5,000 EPS 可通过水平扩展线性提升 |
| **API 性能** | 核心 API 吞吐量达到 10,000 req/s，告警搜索等高频接口 P99 延迟 < 300ms，体验流畅 |
| **并发支持** | 500 并发用户下系统运行稳定，大屏实时刷新与日常操作混合负载无感知延迟 |
| **AI 推理** | DeepSeek 模型单次推理 1.2s，通义千问模型 1.8s，通过批量推理和模型量化可进一步提升吞吐 |
| **数据库性能** | KingbaseES 在 500 万级数据下 P99 查询延迟 200ms，通过索引优化和连接池调优可进一步提升 |
| **实时推送** | 全链路 WebSocket 推送延迟 P99 < 100ms，满足实时告警展示需求 |
| **信创兼容性** | 信创环境整体性能达到 x86 环境的 90%~95%，主要瓶颈主要在 CPU 架构差异和 KingbaseES 兼容层，差异在可接受范围内 |

### 8.3 风险与建议

| 风险项 | 等级 | 说明 | 缓解措施 |
