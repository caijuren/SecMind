**文档版本**: v2.3.0  
**密级**: 客户内部公开  
**适用对象**: 运维团队、系统管理员、技术负责人

## 1. 版本号规则说明

SecMind 平台采用 **语义化版本控制（SemVer）** 规范，版本号格式为 `主版本号.次版本号.修订号`，遵循以下规则：

```
X.Y.Z
│ │ │
│ │ └── 修订号 (Patch)：向后兼容的缺陷修复、安全补丁
│ └──── 次版本号 (Minor)：向后兼容的功能新增、优化
└────── 主版本号 (Major)：不向后兼容的重大变更
```

### 版本号示例与含义

| 版本号 | 含义 | 升级影响 |
|--------|------|---------|
| v2.3.0 | 次版本升级，新增功能 | 向后兼容，建议升级 |
| v2.3.1 | 修订版，Bug修复 | 向后兼容，推荐升级 |
| v3.0.0 | 主版本升级，可能包含Breaking Changes | 需评估兼容性，规划升级窗口 |

### 版本发布生命周期

```
v2.3.0-alpha     ->  内部预览版，仅供测试
v2.3.0-beta      ->  公测版，功能稳定但仍可能有改动
v2.3.0-rc.1      ->  候选发布版，最终测试
v2.3.0           ->  正式发布版（GA）
v2.3.1           ->  修订版（Bug修复）
v2.4.0           ->  下一个功能版本
```

### 版本兼容性矩阵

| 组件 | 版本兼容策略 | 说明 |
|------|-------------|------|
| API 服务 | 次版本内向前兼容 | v2.x API 保证兼容 v2.x 客户端 |
| 数据库 Schema | 次版本内可升降级 | Alembic | 支持升级和回滚 |
| 前端应用 | 需匹配后端主次版本 | 前端 v2.3.x 需配合后端 v2.3.x |
| AI 模型接口 | 独立于平台版本 | 模型API版本变化不影响平台版本 |
| 配置文件 | 次版本兼容 | 新增配置项有默认值 |

## 3. 从v2.x升级到v2.y的通用步骤

### 3.1 升级流程总览

```
+-----------+     +-----------+     +-----------+     +-----------+     +-----------+
| 准备阶段   | --> | 代码更新   | --> | 数据库迁移 | --> | 服务更新   | --> | 验证阶段   |
| 备份/检查  |     | 拉取Tag   |     | Alembic   |     | 拉镜像重启 |     | 功能/性能  |
+-----------+     +-----------+     +-----------+     +-----------+     +-----------+
     |                 |                 |                 |                 |
     v                 v                 v                 v                 v
  时间轴: 0 min      5 min           15 min           25 min           40 min
```

### 3.2 详细操作步骤

```bash
# ============================================

# 注意事项：
#   - 以下命令中的版本号 v2.3.0 需替换为实际目标版本
#   - 如使用 Docker Compose，请确认 compose 文件路径
# ============================================

# ---- 步骤1: 进入部署目录 ----
cd /opt/secmind

# ---- 步骤2: 记录当前状态（用于回滚） ----
echo "=== 升级前状态快照 ===" > /tmp/upgrade_snapshot.txt
git rev-parse HEAD >> /tmp/upgrade_snapshot.txt
docker ps --format "table {{.Names}}\t{{.Image}}\t{{.Status}}\t{{.Ports}}" >> /tmp/upgrade_snapshot.txt
psql -c "SELECT version(), current_timestamp;" >> /tmp/upgrade_snapshot.txt

# ---- 步骤3: 拉取新版本代码（Git部署方式） ----
git fetch --all --tags
git checkout v2.3.0

# 验证代码完整性
git verify-tag v2.3.0 2>/dev/null || echo "标签签名验证未配置，跳过"

# ---- 步骤4: 合并配置文件变更 ----
# 查看 .env.example 是否有新增配置项
diff -u .env.backup.* .env.example | less

# 将新增配置项追加到当前 .env 文件
# 特别注意：新增的配置项可能有默认值，但部分可能需要手动调整
cp .env .env.current
python3 -c "
# 自动合并配置项
import configparser
# 手动合并，保留现有配置同时加入新增项
"

# ---- 步骤5: 执行数据库迁移（关键步骤） ----
# 备份数据库（迁移前再次确认）
pg_dump -h localhost -U secmind -d secmind -F c \
  -f /data/backups/pre_migration_$(date +%Y%m%d_%H%M%S).dump

# 执行 Alembic 迁移
docker compose -f docker-compose.prod.yml run --rm \
  -e DATABASE_URL=postgresql://secmind:password@postgres-primary:5432/secmind \
  api-server alembic upgrade head

# 验证迁移状态
docker compose -f docker-compose.prod.yml run --rm api-server alembic current

# 检查迁移日志是否有异常
docker compose logs --tail=50 api-server | grep -i "migration\|alembic\|error"

# 如果迁移涉及数据迁移（如 backfill），检查数据完整性
psql -c "SELECT count(*) FROM alembic_version;"
psql -c "SELECT version_num FROM alembic_version;"

# ---- 步骤6: 拉取新版本镜像 ----
docker compose -f docker-compose.prod.yml pull

# ---- 步骤7: 优雅重启服务 ----
# 如果有多副本：
# 先停止旧版本 API 服务
docker compose -f docker-compose.prod.yml up -d --no-deps \
  --scale api-server=0 api-server
# 等待旧连接断开
sleep 15
# 启动新版本 API 服务
docker compose -f docker-compose.prod.yml up -d --no-deps \
  --scale api-server=4 api-server

# 如果是单机部署（短暂停机）：
docker compose -f docker-compose.prod.yml down
docker compose -f docker-compose.prod.yml up -d

# 重启 Celery Worker（如果有变更）
docker compose restart celery-worker

# ---- 步骤8: 升级后验证 ----
# 执行升级验证脚本
bash /opt/secmind/scripts verify_upgrade.sh

# 或者手动验证关键功能:
# 1. 健康检查
curl -s http://localhost:8000/api/v1/health | jq .

# 2. 登录
# 验证 API 版本
curl -s http://localhost:8000/api/v1/version | jq .

# 3. 验证数据库连接
docker exec api-server python -c "
```

## 4. 数据库迁移说明

### 4.1 Alembic 迁移流程

SecMind 使用 **Alembic** 进行数据库 Schema 管理，迁移文件存放在 `alembic/versions/` 目录下。

```
alembic/
├── env.py              # 迁移环境配置
├── alembic.ini         # Alembic 配置文件
└── versions/           # 迁移版本文件
    ├── 001_initial_schema.py
    ├── 002_add_ai_analysis.py
    ├── 003_add_rls_policies.py
    ├── 004_add_threat_intel.py
    └── ...
```

### 4.2 自动迁移 vs 手动迁移

| 迁移类型 | 用途 | 自动/手动 | 说明 |
|---------|------|-----------|------|
| Schema 变更 | 新增表、字段、索引 | 自动 | Alembic 自动检测 ORM 模型变更 |
| 数据迁移 | 数据回填、格式转换 | 手动检查 | 需开发人员编写数据迁移逻辑 |
| 索引优化 | 新增/删除复合索引 | 自动生成+手动调整 | 自动生成 SQL 后人工审核 |
| 权限变更 | RLS 策略、角色权限 | 激活函数 | 手动编写 | 需根据业务逻辑手动编写 |
| 种子数据 | 默认配置、系统参数 | 手动执行 | 通过脚本初始化 |

### 4.3 迁移执行命令

```bash
# 查看当前数据库迁移版本
alembic current

# 查看所有迁移历史
alembic history

# 查看待执行的迁移（不执行）
alembic upgrade --sql head

# 执行所有待处理迁移（标准操作）
alembic upgrade head

# 执行到指定版本
alembic upgrade abc123def456

# 回滚一个版本
alembic downgrade -1

# 回滚到指定版本
alembic downgrade abc123def456

# 生成新的自动迁移脚本
alembic revision --autogenerate -m "add_user_preferences_table"

# 标记当前数据库版本（紧急恢复用）
alembic stamp abc123def456
```

### 4.4 迁移过程中的数据验证

```sql
-- 迁移完成后执行数据完整性检查

-- 1. 检查新表是否创建
SELECT table_name FROM information_schema.tables
WHERE table_schema = 'public';

-- 2. 检查新增字段
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_name = 'alerts'
ORDER BY ordinal_position;

-- 3. 检查外键约束
SELECT
    tc.table_name,
    kcu.column_name,
    ccu.table_name AS foreign_table_name,
    ccu.column_name AS foreign_column_name
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
    ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage ccu
    ON ccu.constraint_name = tc.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND tc.table_schema = 'public';

-- 4. 检查索引
SELECT
    tablename,
    indexname,
    indexdef
FROM pg_indexes
WHERE schemaname = 'public'
ORDER BY tablename, indexname;

-- 5. 检查 RLS 策略
SELECT
    tablename,
    policyname,
    permissive,
    roles,
    cmd,
    qual
FROM pg_policies
WHERE schemaname = 'public'
ORDER BY tablename, policyname;

-- 6. 数据一致性检查（示例：检查告警数量级是否合理）
SELECT
    '告警总数' as check_item,
    count(*) as value
FROM alerts
UNION ALL
SELECT
    '今日新增告警',
    count(*)
FROM alerts
WHERE created_at >= CURRENT_DATE
UNION ALL
SELECT
    '待处理案件',
    count(*)
FROM cases
WHERE status IN ('new', 'investigating', 'in_progress');
```

### 4.5 迁移失败处理

对于涉及大量数据迁移的场景（如字段重命名、格式转换），需执行手动验证和异步迁移：

```bash
# 步骤1: 编写数据迁移脚本
# 示例: 将 old_field 迁移到 new_field

# 步骤2: 先在测试环境验证
docker compose -f docker-compose.staging.yml run --rm api-server \
  python scripts/data_migration.py \
  --dry-run \
  --batch-size=1000 \
  --table=alerts

# 步骤3: 在生产环境以小批量执行
docker compose -f docker-compose.prod.yml run --rm api-server \
  python scripts/data_migration.py \
  --batch-size=500 \
  --table=alerts \
  --max-rows=10000 \
  --sleep=1

# 步骤4: 验证迁移结果
psql -c "
SELECT
    'total' as type, count(*) FROM alerts
UNION ALL
SELECT
    'migrated', count(*) FROM alerts WHERE new_field IS NOT NULL;
"

# 步骤5: 分批次完成全部迁移
docker compose -f docker-compose.prod.yml run --rm api-server \
  python scripts/data_migration.py \
  --batch-size=1000 \
  --table=alerts \
  --sleep=0.5
```

## 5. 关键版本间升级注意事项

### 5.1 重点 Breaking Changes 标注

以下列出了各次版本升级中可能影响现有功能的 Breaking Changes，升级前必须仔细阅读。

#### v2.0.x -> v2.1.0 (2024年6月)

**Breaking Changes:**

1. **API URL 变更**：`/api/alerts` 迁移至 `/api/v1/alerts`
   - 影响：所有 API 集成需要更新 Base URL
   - 兼容：旧路径在 v2.x 系列保留可用，v3.0 移除
   - 操作：更新第三方系统的 API Endpoint 配置

2. **认证方式更新**：从 JWT Query Parameter 方式改为仅支持 Bearer Header 方式
   - 影响：使用 `?token=` 参数认证的集成需要修改
   - 兼容：v2.1.x 仍支持，v2.2.x+ 移除
   - 操作：更新集成代码，统一使用 Authorization Header

3. **WebSocket 连接地址变更**：从 `/ws` 改为 `/api/v1/ws/events`
   - 影响：已有 WebSocket 客户端需要更新连接地址
   - 操作：更新前端/集成代码中的 WebSocket URL

#### v2.1.x -> v2.2.0 (2024年10月)

**Breaking Changes:**

1. **告警数据模型扩展**：`alerts` 表新增 `source_id` 和 `raw_data` 必填字段
   - 影响：通过 API 创建告警时需传入新字段，否则返回 422
   - 操作：更新告警推送脚本，增加 `source_id` 和 `raw_data` 字段
   - 参考迁移示例：
     ```json
     // 旧格式
     {
         "title": "告警标题",
         "severity": "high"
     }
     // 新格式
     {
         "title": "告警标题",
         "severity": "high",
         "source_id": "external_alert_001",
         "raw_data": { ... }
     }
     ```

2. **速率限制策略调整**：免费版从 120次/分钟 降至 60次/分钟
   - 影响：免费版用户的 API 调用频率降低
   - 操作：评估是否需要升级套餐或优化调用频率

3. **报表 API 响应格式变更**：`GET /reports/list` 返回结构新增 `pagination` 字段
   - 影响：如直接解析 JSON 数组则不受影响
   - 操作：确认集成代码兼容新响应结构

4. **处置任务状态变更**：状态枚举值规范化
   - 旧值: `pending`, `running`, `success`, `fail`
   - 新值: `pending`, `in_progress`, `completed`, `failed`, `cancelled`
   - 操作：更新状态机逻辑

#### v2.2.x -> v2.3.0 (2025年1月)

**Breaking Changes:**

1. **AI 模型接口响应格式变更**：`POST /ai/analyze/alert` 响应结构调整
   - 旧响应：分析文本放在 `data.result` 字段
   - 新响应：分析结果结构化，包含 `summary`, `root_cause`, `impact_assessment`, `recommendation` 子字段
   - 兼容：新版本同时提供 `data.result`（兼容旧格式）和结构化字段
   - 操作：建议尽快迁移到新的结构化字段以获得更丰富的分析结果

2. **WebSocket 事件格式扩展**：新增 `tenant_id` 字段（多租户场景）
   - 影响：WebSocket 消息解析逻辑需要兼容新字段
   - 操作：更新 WebSocket 客户端消息处理逻辑，忽略未知字段

3. **.env 配置项变更**：以下配置项更名或移除
   - `OPENAI_API_KEY` 更名为 `AI_OPENAI_API_KEY`
   - `AI_MODEL` 拆分为 `AI_PRIMARY_MODEL` 和 `AI_FALLBACK_MODELS`
   - `LOG_LEVEL` 默认值从 `INFO` 改为 `WARNING`

## 6. 回滚方案

### 6.1 回滚决策树

```
升级后发现问题？
├── 数据库迁移已执行但代码未更新完成？
│   └── 直接回滚代码，数据库降级（如有必要）
├── 数据库迁移已执行且发现新版本有问题？
│   ├── 数据迁移不涉及内容变更？
│   │   └── 回滚代码 + 无需数据库回滚
│   └── 涉及字段变更或数据迁移？
│       └── 回滚代码 + 数据库降级 + 数据恢复
├── 新版本运行但性能退化？
│   └── 配置调优尝试 -> 如无效则回滚
└── 新版本运行正常但第三方集成不兼容？
    └── 评估是升级还是等待集成方适配
```

### 6.2 快速回滚（代码回滚）

```bash
# ---- Docker Compose 环境快速回滚 ----

# 方案A：使用镜像标签回滚（推荐）
# 1. 确认上一个正常版本的镜像标签
docker images | grep secmind/api-server

# 2. 重新指向旧版本镜像
docker compose -f docker-compose.prod.yml up -d --no-deps \
  api-server=secmind/api-server:2.2.0

# 3. 验证回滚后状态
curl -s http://localhost:8000/api/v1/health | jq .

# ---- Kubernetes 环境回滚 ----
kubectl rollout undo deployment/api-server -n secmind
kubectl rollout status deployment/api-server -n secmind --timeout=300s

# ---- Git 代码回滚 ----
cd /opt/secmind
git checkout v2.2.0

# 重启服务
docker compose -f docker-compose.prod.yml down
docker compose -f docker-compose.prod.yml up -d
```

### 6.3 数据库回滚

```bash
# ---- 步骤1: 确认当前迁移版本 ----
alembic current

# ---- 步骤2: 查看目标版本（上一个稳定版本） ----
alembic history

# ---- 步骤3: 回滚数据库 Schema ----
# 方式A：逐步回滚到指定版本
alembic downgrade <target_revision>

# 方式B：回滚指定步数
alembic downgrade -1   # 回滚1步
alembic downgrade -2   # 回滚2步

# ---- 步骤4: 验证数据库版本 ----
alembic current
# 确认回滚到升级前的版本

# ---- 步骤5: 数据恢复（如果 Schema 回滚涉及数据丢失） ----
# 如果在升级时创建了新的表或字段导致数据写入：
# 方式A：从备份恢复
pg_restore -h localhost -U secmind -d secmind \
  -F c /data/backups/pre_upgrade_$(date +%Y%m%d).dump \
  --clean --if-exists

# 方式B：恢复到指定时间点（PITR）
# 参考 运维手册.md 的 PITR 恢复流程

# ---- 步骤6: 恢复完成后执行 VACUUM ----
psql -c "VACUUM VERBOSE ANALYZE;"
```

### 6.4 数据库回滚注意事项

| 迁移类型 | 回滚可行性 | 风险 | 说明 |
|---------|-----------|------|------|
| 新增表 | 完全可回滚 | 低 | `downgrade` 自动 DROP TABLE |
| 新增字段 | 完全可回滚 | 低 | `downgrade` 自动 DROP COLUMN |
| 删除字段 | 回滚可恢复 | 中 | 需确认字段被删除前数据已备份 |
| 数据迁移 | 不可自动回滚 | 高 | 需手动编写逆向迁移脚本 |
| 索引变更 | 完全可回滚 | 低 | `downgrade` 自动回退索引变更 |
| 约束变更 | 完全可回滚 | 低 | `downgrade` 自动恢复约束 |

### 6.5 数据一致性检查

数据库回滚完成后，执行数据一致性检查：

```sql
-- 数据库回滚后的一致性检查

-- 1. 行级数据完整性
SELECT schemaname, tablename, n_live_tup, n_dead_tup
FROM pg_stat_user_tables
ORDER BY n_live_tup DESC;

-- 2. 检查外键完整性
SELECT
    conname AS constraint_name,
    conrelid::regclass AS table_name
FROM pg_constraint
WHERE contype = 'f'
AND connamespace = 'public'::regnamespace
AND NOT EXISTS (
    SELECT 1
    FROM pg_class c
    JOIN pg_namespace n ON c.relnamespace = n.oid
    WHERE c.oid = conrelid
);

-- 3. 检查序列是否与数据一致
SELECT
    schemaname || '.' || tablename AS table_name,
    seq_scan,
    seq_tup_read,
    idx_scan,
    idx_tup_fetch
FROM pg_stat_user_tables
ORDER BY seq_scan DESC;

-- 4. 检查 Row Level Security 策略是否恢复
SELECT
    schemaname,
    tablename,
    policyname,
    permissive,
    qual
FROM pg_policies
WHERE schemaname = 'public'
ORDER BY tablename, 2, 3;
```

### 6.6 完整回滚流程脚本

```bash
#!/bin/bash
# /usr/local/bin/rollback.sh
# SecMind 完整回滚脚本 - v2.3.0 -> v2.2.0

set -e

echo "[$(date)] === 开始回滚流程 ==="

# 配置
TARGET_VERSION="v2.2.0"
BACKUP_DIR="/data/backups"
PREVIOUS_BACKUP=$(ls -t ${BACKUP_DIR}/pre_upgrade_*.dump 2>/dev/null | head -1)

# 检查备份文件
if [ -z "${PREVIOUS_BACKUP}" ]; then
    echo "错误：未找到升级前备份文件，请手动指定"
    echo "用法: $0 <备份文件路径>"
    exit 1
fi

echo "使用备份文件: ${PREVIOUS_BACKUP}"

# 步骤1: 停止服务
echo "[步骤1] 停止当前服务..."
docker compose -f docker-compose.prod.yml down

# 步骤2: 回滚代码
echo "[步骤2] 回滚代码到 ${TARGET_VERSION}..."
cd /opt/secmind
git checkout ${TARGET_VERSION}

# 步骤3: 回滚配置文件
echo "[步骤3] 恢复配置文件..."
cp .env.backup.* .env 2>/dev/null || echo "备份不存在，手动检查 .env 文件"

# 步骤4: 回滚数据库 Schema
echo "[步骤4] 回滚数据库 Schema..."
# 启动数据库（仅）
docker compose -f docker-compose.prod.yml up -d postgres-primary
sleep 15

# 执行 Schema 降级
docker compose -f docker-compose.prod.yml run --rm api-server \
  alembic downgrade <target_revision>
# 注意: <target_revision> 需替换为升级前的 revision 编号

# 如果 Schema 降级不成功，从备份恢复
if [ $? -ne 0 ]; then
    echo "Schema 降级失败，从备份恢复..."
    pg_restore -h localhost -U secmind -d secmind \
      -F c "${PREVIOUS_BACKUP}" --clean --if-exists
fi

# 步骤5: 恢复旧版本镜像并启动
echo "[步骤5] 启动旧版本服务..."
docker compose -f docker-compose.prod.yml up -d

# 步骤6: 验证
echo "[步骤6] 验证回滚结果..."
sleep 20
curl -s http://localhost:8000/api/v1/health | jq .
curl -s http://localhost:8000/api/v1/version | jq .

# 基础功能测试
echo "执行基础功能测试..."
python3 /opt/secmind/scripts/quick_test.py

echo "[$(date)] === 回滚完成 ==="
echo "请执行完整的升级后验证（第7章）"
```
