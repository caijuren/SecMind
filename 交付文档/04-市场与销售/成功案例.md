## 二、案例一：某股份制商业银行

### 案例名称：某股份制商业银行SecMind AI安全运营平台实践

### 客户背景

- **行业**：金融 — 股份制商业银行
- **企业规模**：总资产约1.2万亿元，全国设有分行30余家，员工总数逾万人
- **IT资产规模**：托管服务器2,800+台，网络设备1,500+台，安全设备400+台，终端设备12,000+个
- **安全团队规模**：10人（含SOC经理1人、Tier-3分析师2人、Tier-2分析师3人、Tier-1分析师4人）
- **所属区域**：华东地区

### 业务挑战

**挑战一：告警量激增，团队不堪重负**

该银行日均原始告警量超过15,000条，高峰时期可达30,000条/天。告警来源涵盖NGFW、IPS、WAF、WAF、IPS、EDR、邮件安全、数据库审计等15类安全设备。传统SIEM系统基于规则引擎，缺乏有效的告警降噪能力，实际可研判的告警比例不足20%，大量真实威胁被淹没在告警洪流中。

**挑战二：分析师告警疲劳严重**

Tier-1分析师每天需要手动标记和处理300-500条告警，其中80%以上为误报。高强度重复劳动导致分析师流失率高达30%（行业平均25%），新人培养需要6-12个月，团队战斗力持续下降。

**挑战三：MTTR过长，安全事件响应滞后**

从告警触发到完成处置的平均时间（MTTR）长达4-6小时。对于Webshell上传、勒索软件投递等高危事件，数小时的响应窗口足以造成实质性损害。安全团队急需缩短响应时间。

**挑战四：信创合规要求**

作为股份制银行，需满足等保2.0三级要求，同时监管机构对金融行业信创替代提出明确时间表。原有SIEM基于国外开源组件，在信创适配方面存在合规风险。

### 选型过程

该银行在2024年Q3启动了新一代安全运营平台的选型工作，历时3个月，评估了6家厂商：

| 厂商 | 评估结论 |
|------|----------|
| 奇安信天眼 | AI能力弱，部署周期长，与现有环境集成复杂 |
| 绿盟 | AI能力不足，产品体验老旧，不符合行内对AI能力的预期 |
| 深信服 | 产品深度不足，大型银行案例少 |
| Splunk ES | 信创不支持，按数据量计费成本不可控，年费超500万 |
| IBM QRadar | 信创不支持，架构老旧，运维成本高 |
| **SecMind（最终选择）** | AI能力明显领先，信创适配完整，PoC效果超出预期（告警降噪99%），部署周期短，性价比高 |

### 解决方案

- **部署模式**：私有化部署（符合金融行业合规要求）
- **部署架构**：信创环境 — 鲲鹏920处理器 + 麒麟V10操作系统 + 达梦数据库
- **覆盖范围**：2,500个日志源，日均处理日志量约80GB
- **实施周期**：2周完成部署，4周完成调优上线
- **核心功能**：
  - AI告警引擎（全量告警接入与智能降噪）
  - 自动化处置引擎（预置80+金融行业处置剧本）
  - ChatSec AI助手（安全分析师日常查询与报告生成）
  - 合规报告自动生成（等保2.0三级）

### 实施效果

|------|--------|--------|----------|
| 日均告警量 | 15,000-30,000条 | 150-300条（降噪后需关注量） | 99%降噪率 |

- **覆盖范围**：全省60+个委办局，托管业务系统500+个，承载数据量超5PB
- **IT资产规模**：政务云平台物理服务器1,200+台，虚拟机6,000+台，安全设备200+台
- **安全团队规模**：20人（含政务云安全运营中心SOC团队15人+各委办局安全联络员5人）
- **所属区域**：华中地区

### 业务挑战

**挑战一：等保2.0合规要求严格**

根据等保2.0三级要求，政务云平台必须建设统一的安全运营中心，实现全面的日志采集、关联分析、告警管理、应急处置和合规报告能力。同时需满足《政务云安全管理办法》《数据安全法》等多重合规要求。

**挑战二：多租户场景复杂**

政务云平台服务于60+个委办局，各部门的数据安全要求、资产归属、权限管理各不相同。需要一个支持多租户、数据隔离、分级授权的安全管理平台。

**挑战三：信创替代刚性要求**

作为政府信息化基础设施，政务云平台要求在2025年底前完成全栈信创替代。国外SOC产品（Splunk/IBM QRadar）被明确列入禁止采购清单，原有安全运营系统需要切换至全面国产化方案。

政务云日均产生安全告警约25,000条，误报率超过85%。20人的安全团队疲于应付海量告警，缺乏有效手段进行威胁研判和处置。

**挑战五：安全运维人员技能参差不齐**

各委办局的安全联络员多为兼职，安全分析能力有限。需要一个低上手门槛、AI辅助的智能运营平台，降低对人员技能的依赖。

## 四、案例三：某大型企业集团

### 案例名称：某大型企业集团子公司统一安全管理实践

### 客户背景

- **行业**：多元化企业集团（能源+制造+地产+金融） — 大型央企集团
- **企业规模**：世界500强，年营收超2,000亿元，员工总数超10万人
- **组织架构**：集团总部 + 8大业务板块 + 50+家子公司
- **IT资产规模**：50+家子公司，每家子公司IT资产规模不等（小型200-中型2,000台）
- **安全团队现状**：集团总部安全团队10人，各子公司兼职安全运维人员1-3人（多为IT运维兼任）
- **所属区域**：华北地区（集团总部北京，子公司分布全国）

### 业务挑战

**挑战一：多子公司安全能力参差不齐**

50+家子公司中，仅10家大型子公司有专职安全人员，其余40+家子公司由IT运维兼任安全职责。各子公司的安全能力、流程规范、工具选型各不相同，集团总部无法掌握各子公司的真实安全状况。

**挑战二：安全建设重复投入**

各子公司独立采购安全设备和工具，导致全集团安全投入分散且重复。据测算，各子公司每年安全投入合计约2,000万元，但整体安全效果远低于集中化投入的预期。

**挑战三：缺乏统一的安全运营平台**

集团总部缺乏一个能够跨子公司统一管理的安全运营平台，无法实现：
- 统一的安全告警采集和监控
- 标准化的事件响应流程
- 集团级的安全态势感知
- 合规监管的统一上报

**挑战四：合规与审计压力**

作为央企，面临严格的上级监管和审计要求。需要能够汇总展示全集团的安全运营状况，定期生成集团级安全报告，但现有的分散管理模式导致数据收集困难、报告质量参差不齐。

**挑战五：信创替代要求**

作为央企，需在2025年底完成全面信创替代。各子公司原有安全工具品牌不一，信创适配进度缓慢。

### 解决方案

- **部署模式**：集团总部私有化集中部署，各子公司通过统一平台接入
- **部署架构**：信创环境 — 飞腾S2500 + 统信UOS + 人大金仓数据库
- **覆盖范围**：集团总部 + 50+家子公司，统一接入安全日志
- **实施周期**：4周完成集团总部部署，后续8周分批接入各子公司
- **核心功能**：
  - 集团级多租户管理（50+子公司作为独立租户）
  - AI告警引擎（统一接入各子公司安全设备日志）
  - 自动化处置剧本（预置120+集团安全场景剧本）
  - 集团级安全态势大屏（总部管理视角）
  - 子公司自助报表平台（各租户独立报表）

### 6.1 案例文件命名规则

### 6.2 案例敏感信息处理

| 信息类型 | 处理方式 | 说明 |
|----------|----------|------|
| 客户全称 | 模糊化处理 | 除非客户书面授权，否则使用"某XX银行"代替具体名称 |
| 具体数据 | 可公开数据 | 仅展示经客户确认可公开的数据指标 |
| 个人姓名 | 化名处理 | 使用"张行长（化名）"格式，真实姓名不出现在公开案例中 |
| 财务数据 | 范围展示 | 使用"约XX万元""XX%+"等范围形式，避免精确数字 |
| 技术架构 | 概要展示 | 展示架构逻辑，不涉及具体IP、端口等敏感信息 |

### 6.3 案例审批流程

1. 编写案例初稿（SecMind市场部）
2. 客户审核确认（客户信息安全部/市场部）
