# POC标准化流程指南

## 一、POC目的与适用范围

### 1.1 目的

POC（Proof of Concept，概念验证）是在正式采购前，向客户证明SecMind平台能够满足其安全运营需求的关键环节。通过标准化的POC流程，确保：
- 客户能够在真实或模拟环境中验证产品的功能与性能
- 销售团队和售前工程师有明确的执行标准，提升POC成功率
- 减少因准备不充分或执行不规范导致的POC失败
- 为后续商务谈判提供客观的技术依据

### 1.2 适用范围
本流程适用于所有需要进行POC的SecMind产品项目，包括：
- 新客户首次接触SecMind产品
- 现有客户新增功能模块或升级场景
- 竞争性POC（与竞品同台对比）
- 关键行业或大型客户的深度验证

### 1.3 相关角色与职责

| 角色 | 职责 |
|------|------|------|
| 销售经理 | 统筹POC整体节奏，协调客户资源，推动商务进程 |
| 售前工程师 | 技术主导，负责环境搭建、测试执行、报告撰写 |
| 客户对接人 | 提供业务需求、测试环境、数据样本，参与测试评估 |
| 产品经理（必要时） | 处理POC中发现的重大需求或产品缺陷 |
| 研发团队（必要时） | 协助解决POC期间的特殊技术问题 |

## 二、POC准备阶段

### 2.1 环境准备清单

#### 2.1.1 部署环境要求

| 检查项 | 规格要求 | 备注 |
|--------|---------|------|
| 服务器数量 | 最小3节点（管理节点+工作节点） | 生产环境建议5节点以上 |
| 操作系统 | CentOS 7.9+ / Ubuntu 20.04+ / 信创操作系统 | 需提前确认操作系统版本 |
| CPU | 最低16核，建议32核以上 | 取决于数据量 |
| 内存 | 最低64GB，建议128GB以上 | |
| 磁盘 | SSD 2TB以上（数据盘建议NVMe） | 日志存储量需按每日数据量估算 |
| 网络 | 千兆以上内部网络，与安全设备互通 | 需确认与各数据源的网络连通性 |
| Docker环境 | Docker 24.0+ / containerd | 私有化部署必选 |
| Kubernetes集群（可选） | K8s 1.24+ | 容器化部署必选 |
| 浏览器 | Chrome 90+、Firefox 90+、Edge 90+ | 前端访问 |
| HTTPS证书 | 有效TLS证书（测试环境可用自签名证书） | 用于安全通信 |

#### 2.1.2 数据源对接环境

| 检查项 | 说明 |
|--------|------|
| 防火墙策略 | 开放SecMind平台与各数据源的通信端口（Syslog 514/UDP、API 443/TCP、JDBC 3306等） |
| 日志源清单 | 确认客户已有的日志源类型和数量（防火墙、IDS/IPS、EDR、云安全日志等） |
| 日志格式确认 | 确认日志格式（Syslog、JSON、CEF、LEEF等），提前准备解析规则 |
| API密钥权限 | 获取各安全设备的API只读或读写权限 |
| 网络代理配置 | 如客户环境需代理出网，提前配置 |
| 时间同步 | NTP服务确认，确保所有设备时间同步 |

## 三、POC执行阶段

### 3.1 标准测试场景列表

#### 场景1：告警接入测试

**测试目的**：验证SecMind平台能否稳定接入多种来源的安全告警与日志数据。

**前置条件**：
- 已完成环境部署和数据源网络连通性确认
- 已获取数据源基本信息（IP、端口、协议、凭证）

**测试步骤**：

| 步骤  | 操作描述 | 预期结果 |
|-------|---------|---------|
| 1 | 在SecMind平台添加数据源配置（Syslog、API、JDBC等） | 数据源连接状态显示"正常" |
| 2 | 推送测试日志/告警到平台 | 平台实时接收到日志，日志解析无异常 |
| 3 | 验证日志标准化效果 | 原始日志被正确解析为标准字段（源IP、目的IP、事件类型、时间等） |
| 4 | 持续接入24小时数据 | 数据接入稳定无中断，无丢数据 |
| 5 | 验证日志检索功能 | 按时间、IP、事件类型等条件检索，结果准确 |
| 6 | 验证数据完整性（原始日志数vs平台接入数对比） | 数据完整度≥99.9% |

**通过标准**：数据源稳定接入，日志解析正确率≥99%，检索功能正常。

---

#### 场景2：AI分析准确率测试

**测试目的**：验证SecMind AI引擎对安全告警的分析降噪和误报判断能力。

**前置条件**：
- 已完成至少3天的告警数据接入
- AI模型已完成基线学习

**测试步骤**：

| 步骤 | 操作描述 | 预期结果 |
|------|---------|---------|
| 1 | 选取客户过去7天原始告警数据（不少于10,000条） | 确认数据已导入平台 |
| 2 | 人工标注其中真实告警和误报告警（建立Ground Truth） | 标注完成，作为对照标准 |
| 3 | AI引擎自动分析告警并输出分析标签 | AI对每条告警给出风险评分和分类 |
| 4 | 对比AI分析结果与人工标注结果 | 生成准确率、召回率、F1-Score指标 |
| 5 | 检查AI对告警的富化信息（关联上下文、资产信息等） | 告警信息完整，包含上下文关联 |
| 6 | 验证AI聚类/聚合效果 | 同类告警被自动聚合，减少重复 |

**通过标准**：
- 告警降噪率≥70%（误报告警被有效过滤）
- 真实告警检出率（召回率）≥95%
- 高风险告警零漏报
- 告警富化字段完整率≥90%

---

#### 场景3：自动处置测试

**测试目的**：验证SecMind平台通过剧本（Playbook）对安全事件进行自动化处置的能力。

**前置条件**：
- 已完成至少一个数据源的读写API对接
- 已配置2-3个标准处置剧本

**测试骤**：

| 步骤 | 操作描述 | 预期结果 |
|------|---------|---------|
| 1 | 查看预置剧本列表 | 展示剧本名称、触发条件、动作列表 |
| 2 | 选择一个预置剧本（如"疑似勒索软件自动隔离"） | 确认剧本配置完整 |
| 3 | 模拟触达完整执行链 | 剧本自动执行，各步骤状态标记为"完成" |
| 4 | 验证处置动作执行情况（如防火墙阻断IP） | 目标设备上确认策略已生效 |
| 5 | 验证处置回滚功能 | 回滚操作后，被阻断策略自动撤销 |
| 6 | 验证自定义剧本创建 | 从零创建一个简单剧本，保存并启用 |

**通过标准**：
- 剧本整体执行成功率≥95%
- 单一处置动作执行时间≤30秒
- 回滚操作100%可逆
- 支持人工审批节点

#### 场景4：威胁狩猎测试

**测试目的**：验证SecMind平台的主动威胁发现和威胁狩猎能力。

**前置条件**：
- 已接入至少3种以上日志源
- 已加载威胁情报数据

**测试步骤**：

| 步骤 | 操作描述 | 预期结果 |
|------|---------|---------|
| 1 | 使用内置威胁狩猎模板进行搜索 | 模板内置≥20个标准狩猎规则 |
| 2 | 按已知IOC（IP/域名/Hash）进行关联搜索 | 精准匹配到相关日志记录 |
| 3 | 使用MITRE ATT&CK框架进行行为分析 | 识别出与已知TTP匹配的攻击行为 |
| 4 | 创建自定义狩猎规则 | 自定义规则能正常运行并产生预期结果 |
| 5 | 验证异常行为检测（如非工作时间登录、异常流量等） | 识别出异常行为并生成告警 |
| 6 | 验证狩猎结果的可导出性 | 支持CSV/PDF/JSON格式导出 |

**通过标准**：
- IOC匹配准确率≥95%
- 支持MITRE ATT&CK v14+框架映射
- 狩猎结果响应时间≤5秒
- 自定义规则创建流程完成可执行

- 预置报表模板覆盖等保2.0、ISO27001、PCI-DSS等

- 定时发送功能正常

## 五、POC周期参考

### 5.1 标准POC周期（2周）

| 阶段 | 时间 | 主要工作 |
|------|------|---------|
| 第1-2天 | | 客户需求调研、环境确认、环境部署 |
| 第3-5天 | 数据接入 | 日志源对接、数据接入、解析验证 |
| 第6-8天 | 功能验证 | AI分析、案件管理、自动处置测试 |
| 第9-10天 | 场景验证 | 威胁狩猎、合规报表、大屏展示 |
| 第11天 | 性能测试 | 压力测试、稳定性验证 |
| 第12天 | 报告撰写 | 整理测试数据、撰写POC报告 |
| 第13-14天 | 汇报与评审 | POC汇报会、客户评估、商务推进 |

### 5.2 复杂POC周期（4周）

| 阶段 | 时间 | 主要工作 | 说明 |
|------|------|---------|------|
| 第1周 | 环境搭建与基础对接 | 私有化部署、多数据源对接、网络适配 |
| 第2周 | 功能深度验证 | 全功能模块测试、定制化场景验证 |
| 第3周 | 性能压测与稳定性 | 长时间稳定性测试（72h+）、压力测试 | 并行演示汇报 |
| 第4周 | 报告与商务推进 | POC报告完善、客户评审会、商务谈判 |

### 5.3 周期影响因素

| 因素 | 影响 | 应对措施 |
|------|------|---------|
| 客户网络环境复杂（多分区、策略严格） | 部署和对接时间增加3-5天 | 提前远程巡检环境 |
| 数据源种类多（10+类日志源） | 接入工作量增加 | 批量接入方案，标准格式优先 |
| 客户需求频繁变更 | 测试范围蔓延 | 严格控制变更，签订需求确认函 |
| 客户配合度低 | 整体进度延迟 | 商务层面推动，明确时间节点责任 |
| 涉及信创环境适配 | 需额外3-5天适配测试 | 提前确认操作系统和CPU架构 |

## 八、POC失败处理

### 8.1 失败原因分类

| 失败类型 | 典型原因 | 占比估计 |
|---------|---------|---------|
| 产品功能不足 | 缺少客户核心需求功能，或AI准确率未达标 | 30% |
| 部署对接失败 | 客户环境过于特殊，无法完成部署或对接 | 15% |
| 性能不达标 | 吞吐量、响应时间等无法满足客户要求 | 20% |
| 竞品对比劣势 | 与竞品对比时在某些维度表现不足 | 15% |
| 客户关系问题 | 客户已有倾向性选择，POC仅为走流程 | 10% |
| 准备不充分 | 测试环境或数据准备不足导致效果不佳 | 10% |

### 8.2 失败后的跟进策略

#### 8.2.1 产品功能不足时的处理

1. **区分必选和期望**：确认客户提到的功能属于"必须有"还是"最好有"
2. **产品路线图支撑**：如功能在后续版本规划中，向客户展示Roadmap
3. **定制开发评估**：对重要客户评估定制开发可行性（需研发评估周期）
4. **变通方案**：使用现有功能组合是否能达成相似效果
5. **记录反馈**：将客户需求正式录入产品反馈系统，持续跟踪

#### 8.2.2 部署对接失败时的处理

1. **技术评估升级**：安排更资深的研发工程师介入分析
2. **替代方案**：评估SaaS方案是否可替代私有化部署
3. **中间件辅助**：通过Logstash/Fluentd等中间件对接
4. **网络策略协调**：协调客户IT部门调整网络策略
5. **远程vs现场**：必要时安排研发人员驻场解决

#### 8.2.3 性能不达标时的处理

1. **瓶颈分析**：定位性能瓶颈（网络、磁盘IO、CPU、内存）
2. **配置优化**：调整系统参数，优化资源分配
3. **横向扩展**：增加节点数，提升整体处理能力
4. **数据采样**：如全量处理无法满足，评估采样方案可行性
5. **分级处理**：高风险告警全量分析，低风险告警抽样分析

#### 8.2.4 竞品对比劣势时的处理

1. **差异化优势**：突出SecMind在AI分析、全链路自动化等方面的独特优势
2. **TCO对比**：核算总拥有成本（许可+部署、运维人员成本）
3. **开放生态**：强调API开放性和集成能力
4. **互操作性**：展示与客户已有系统的兼容性（非替换）
5. **长期演进**：展示产品规划和技术路线图的领先性

### 8.3 失败后续跟进流程

```
POC未通过
  │
  ▼
失败原因分析（48小时内完成）
  │  • 内部复盘会，形成分析报告
  ▼
制定改善计划
  │  • 明确改善内容和技术负责人
  │  • 预估改善完成时间
  ▼
与客户沟通
  │  • 正视问题，展示改善计划
  │  • 通报改善计划和时间表
  │  • 保持商务关系，不放弃
  ▼
改善完成
  │  • 内部验证改善
  ▼
再约POC
  │  • 确定合理的再POC时间（至少间隔1-2个月）
  │  • 针对上次失败点重点验证
  ▼
再次执行POC流程
```

### 8.4 失败后的客户关系维护

- **定期沟通**：每2-4周同步一次产品改进进展
- **邀请参加活动**：客户活动、行业峰会、产品发布会
- **资源共享**：持续输出安全运营白皮书、行业报告等价值内容
- **技术交流**：保持技术层面的定期交流，展示团队专业能力
- **分期关注**：关注客户的采购计划变化，选择合适时机再次切入

## 九、常见POC风险与应对

### 9.1 风险清单

| 风险类别 | 风险描述 | 概率 | 影响 | 应对措施 |
|---------|---------|------|------|---------|
| 技术风险 | 客户环境与标准部署环境差异大 | 中 | 高 | 提前远程巡检，确认环境兼容性 |
| 技术风险 | 数据源格式特殊，标准解析规则无法覆盖 | 中 | 中 | 准备至少2周的解析规则开发和调试时间 |
| 技术风险 | 数据量远超预期，超出平台处理能力 | 低 | 高 | 提前确认日均数据量，预留弹性扩展 |
| 进度风险 | 客户配合度低，数据提供延迟 | 高 | 中 | 商务层面明确时间节点，必要时签订POC协议 |
| 进度风险 | 客户需求频繁变更，测试范围扩大 | 中 | 中 | 严格控制变更，签订需求确认函 |
| 人员风险 | 客户关键决策人变动 | 低 | 高 | 建立多维度客户关系，不依赖单一联系人 |
| 竞争风险 | 竞品同期开展POC | 中 | 高 | 突出差异化优势，展示差异化优势 |
| 商务风险 | 客户仅以POC获取免费试用为目的 | 中 | 中 | 前置评估客户采购意向和预算 |
| 安全风险 | 客户要求在敏感环境部署 | 低 | 高 | 签订保密协议，限制数据访问权限 |
| 安全风险 | POC数据泄露风险 | 低 | 高 | 数据脱敏，最小权限原则，POC后清理数据 |

### 9.2 风险应对矩阵

| 风险等级 | 定义 | 应对策略 | 决策要求 |
|---------|------|-----------------|
| 高风险 | 可能导致POC完全失败 | 需销售总监+技术总监联合评估是否继续 |
| 中风险 | 可能影响POC效果或进度 | 需制定专项应对计划并指定负责人 |
| 低风险 | 影响可控，不影响整体POC | 正常监控，常规应对 |

### 9.3 POC准入条件

在启动POC前，应确认以下基础条件是否满足。不满足时，建议暂缓POC：

1. **客户有明确采购计划**：客户已确认有年度安全预算或近期采购计划
2. **关键决策人参与**：客户安全负责人或技术决策者明确支持POC
3. **技术环境基本匹配**：客户技术环境与SecMind兼容性经过初步评估
4. **客户愿意投入意愿**：客户愿意投入人员和时间配合POC
5. **竞争对手清晰**：明确了解是否存在竞品及竞品情况
6. **POC目标明确**：客户有清晰的POC成功标准和评估维度

### 9.4 POC停止标准

出现以下情况时，应及时中止POC并与客户沟通：

1. **重大不兼容**：发现根本性技术不兼容，短期内无法解决
2. **客户配合中断**：客户连续5个工作日内客户无实质配合动作
3. **关键需求无法满足**：客户核心需求在当前版本完全无法实现
4. **采购计划取消**：客户明确表示采购计划推迟或取消
5. **保密风险升级**：涉及客户敏感数据风险不可控
