# SecMind AI安全运营平台 — 管理员培训指南

**版本**：v2.3.0  
**适用对象**：系统管理员、运维人员  
**预计培训时长**：3天（理论2天 + 实操1天）

---

## 培训目标与前提条件

### 培训目标

完成本培训后，学员将能够：

1. 独立完成SecMind平台的部署安装（单机/高可用/信创环境）
2. 掌握系统配置和维护的核心操作
3. 熟练管理用户、角色和租户
4. 掌握AI模型的配置和调优方法
5. 能够进行日常监控系统健康状态并及时处理异常
6. 掌握备份恢复和版本升级操作
7. 能够排查和解决常见的系统故障

### 前提条件

学员应具备以下基础知识和技能：

- **操作系统**：熟悉Linux操作系统基本操作（CentOS 7/8、Ubuntu 20.04+、麒麟V10等）
- **网络基础**：了解TCP/IP协议、DNS、负载均衡、防火墙等基本网络概念
- **容器技术**：了解Docker和Kubernetes基本概念
- **数据库**：了解PostgreSQL/MySQL等关系型数据库基本操作
- **安全基础**：了解基本的安全运营概念（告警、事件、案件等）

### 培训环境要求

- 一台4核16GB以上配置的Linux服务器（用于实践操作）
- 已获取SecMind v2.3.0安装包和License
- 具备root或sudo权限
- 能够访问互联网（用于下载依赖组件）

## 模块1：系统部署与初始化

### 1.1 部署架构概览

SecMind平台由以下核心服务组件构成：

| 组件 | 功能 | 部署方式 |
|------|------|---------|
| **secmind-web** | Web前端控制台 | Docker/K8s |
| **secmind-api** | 后端API服务 | Docker/K8s |
| **secmind-engine** | AI推理引擎 | Docker/K8s（需GPU） |
| **secmind-worker** | 异步任务处理器 | Docker/K8s |
| **secmind-collector** | 告警采集器 | Docker/裸金属 |
| **secmind-scheduler** | 剧本调度引擎 | Docker/K8s |
| **PostgreSQL** | 业务数据库 | 独立部署/高可用 |
| **Redis** | 缓存与消息队列 | 独立部署/高可用 |
| **Elasticsearch** | 告警存储与搜索引擎 | 集群部署 |
| **MinIO/S3** | 对象存储（证据、附件） | 独立部署 |

### 1.2 单机部署

**操作步骤：**

1. **环境准备**
   ```bash
   # 检查系统要求
   cat /etc/os-release
   uname -m
   
   # 安装依赖
   yum install -y docker docker-compose git wget curl
   systemctl enable docker && systemctl start docker
   
   # 检查GPU驱动（如使用GPU推理）
   nvidia-smi
   ```

2. **获取安装包**
   ```bash
   # 解压安装包
   tar -zxvf secmind-v2.3.0.tar.gz
   cd secmind-v2.3.0
   ```

3. **配置环境变量**
   ```bash
   cp .env.example .env
   vi .env
   # 修改关键配置项：
   # SECMIND_LICENSE=your_license_key
   # DB_PASSWORD=your_secure_password
   # REDIS_PASSWORD=your_redis_password
   # SMTP_HOST=smtp.yourcompany.com
   ```

4. **执行部署脚本**
   ```bash
   ./deploy.sh --mode single
   # 等待所有服务启动完成，约10-15分钟
   ```

5. **验证部署**
   ```bash
   # 检查所有服务状态
   ./secmind-cli status
   
   # 查看服务日志
   docker-compose logs -f secmind-api
   ```

6. **访问平台**
   - 浏览器访问 `http://<服务器IP>:8080`
   - 默认管理员账号：`admin`
   - 默认密码：`Admin@123456`（首次登录强制修改）

### 1.3 高可用集群部署

**集群架构要求：**

- 至少3个Master节点（K8s控制平面）
- 至少3个Worker节点（运行服务）
- 至少3个Elasticsearch节点
- PostgreSQL主从复制 + Pgpool-II负载均衡
- Redis Sentinel或Redis Cluster
- Nginx/Haproxy负载均衡器

**操作步骤：**

1. **搭建Kubernetes集群**
   ```bash
   # 在所有节点上执行
   yum install -y kubelet kubeadm kubectl
   systemctl enable kubelet
   
   # 在Master节点初始化Master节点上执行
   kubeadm init --pod-network-cidr=10.244.0.0/16
   
   # 安装CNI网络插件（Calico）
   kubectl apply -f https://docs.projectcalico.org/manifests/calico.yaml
   ```

2. **配置存储**
   ```bash
   # 部署分布式存储（Longhorn或Ceph）
   kubectl apply -f https://raw.githubusercontent.com/longhorn/longhorn/master/deploy/longhorn.yaml
   ```

3. **部署SecMind集群**
   ```bash
   cd secmind-v2.3.0
   ./deploy.sh --mode cluster --kubeconfig ~/.kube/config
   ```

4. **配置Ingress和负载均衡**
   ```bash
   kubectl apply -f ingress/nginx-ingress.yaml
   kubectl apply -f ingress/secmind-ingress.yaml
   ```

5. **扩容Worker节点**
   ```bash
   kubectl scale deployment secmind-worker --replicas=5
   kubectl scale deployment secmind-engine --replicas=3
   ```

### 1.4 信创环境部署

**信创环境特殊注意事项：**

| 组件 | 信创替代方案 | 注意事项 |
|------|------------|---------|
| 操作系统 | 麒麟V10 / 统信UOS | 需使用ARM64或LoongArch架构版本 |
| CPU | 鲲鹏920 / 飞腾S2500 / 龙芯3A5000 | 确认是否支持硬件虚拟化 |
| GPU（可选） | 华为昇腾310/910 / 寒武纪MLU | 需安装对应驱动和CANN工具包 |
| 数据库 | 达梦DM8 / 人大金仓KingbaseES | 需单独安装JDBC驱动 |
| 容器 | 支持Docker（ARM版） | 部分镜像需重新编译ARM版本 |

**信创部署操作：**

```bash
# 安装ARM版Docker
yum install -y docker-ce-19.03.9-3.el7.aarch64

# 导入信创专用镜像
docker load -i secmind-images-arm64-v2.3.0.tar

# 使用信创部署模式
./deploy.sh --mode single --arch arm64 --db dm8
```

## 模块2：系统配置

### 2.1 系统参数配置

登录管理员后台后，进入「系统设置」→「系统参数」菜单，配置以下关键参数：

**基础设置：**

| 参数 | 说明 | 推荐值 |
|------|------|-------|
| 系统名称 | 平台显示名称 | SecMind-公司名称 |
| 时区 | 系统时区 | Asia/Shanghai |
| 语言 | 默认语言 | zh-CN |
| 会话超时 | 用户登录超时时间（分钟） | 480 |
| 最大登录尝试 | 密码错误锁定阈值 | 5次 |
| 密码策略 | 密码复杂度要求 | 大写+小写+数字+特殊字符，长度≥10 |

**告警处理参数：**

| 参数 | 说明 | 推荐值 |
|------|------|-------|
| 告警归并窗口 | 同类告警归并的时间窗口（秒） | 300 |
| AI分析超时 | AI单条告警分析超时（秒） | 60 |
| 最大待处理告警 | 积压告警上限 | 1,000,000 |
| 自动归档天数 | 告警自动归档天数 | 90 |

### 2.2 邮件服务器配置

**操作步骤：**

1. 进入「系统管理」→「通知配置」→「邮件服务器」
2. 填写以下信息：
   - SMTP服务器：`smtp.yourcompany.com`
   - 端口：`465`（SSL）/ `587`（TLS）
   - 发件人地址：`secmind@yourcompany.com`
   - 发件人名称：`SecMind平台`
   - 认证方式：用户名密码 / OAuth2
3. 点击「测试连接」，系统会发送测试邮件至当前管理员邮箱
4. 确认收到测试邮件后保存配置

**邮件模板配置：**

- 系统提供预置模板：告警通知、案件通知、系统告警、密码重置等
- 每个模板均支持自定义标题和正文内容
- 支持使用占位变量：`${alert_name}`、`${severity}`、`${timestamp}`、`${case_id}`等

### 2.3 告警源接入配置

**通用接入步骤：**

1. 进入「数据接入」→「数据源管理」→「新增数据源」
2. 选择数据源类型（如Splunk、CrowdStrike、Syslog等）
3. 填写连接参数：
   - 名称：数据源的标识名称
   - 类型：自动带出
   - 连接地址：数据源的IP和端口
   - 认证凭据：API Token/用户名密码（视类型而定）
   - 数据过滤规则：可选，按条件过滤无效告警
4. 点击「测试连接」验证连通性
5. 配置完成，状态显示为"已连接"

**Syslog接入示例：**

```bash
# 在数据源配置中选择"Syslog"
# 配置参数：
监听端口: 514
协议: UDP/TCP
日志格式: CEF / LEEF / JSON / Syslog-RFC5424

# 在防火墙上开放端口
firewall-cmd --add-port=514/udp --permanent
firewall-cmd --reload
```

**数据源健康检查：**

- 配置完成后，系统每5分钟自动检测数据源连接状态
- 断开超过10分钟，系统生成「数据源中断」告警
- 可在「数据源管理」页面查看各数据源实时状态

## 模块3：用户与权限管理

### 3.1 用户管理

**创建用户：**

1. 进入「系统管理」→「用户管理」→「新增用户」
2. 填写用户信息：
   - 用户名：字母数字组合，唯一
   - 姓名：用户真实姓名
   - 邮箱：必填，用于接收通知
   - 手机号：可选，用于短信通知
   - 所属部门：可选，用于组织架构展示
   - 角色：选择一个或多个角色
   - 初始密码：可自动生成或手动设置
   - 密码过期时间：设置密码有效期（默认90天）
3. 点击「保存」，系统发送账号开通通知至用户邮箱

**批量导入用户：**

1. 在用户管理页面点击「批量导入」
2. 下载CSV模板
3. 按模板格式填写用户信息（支持50个用户/批次）
4. 上传CSV文件并确认
5. 系统校验数据格式，成功则批量创建用户

**用户状态管理：**

| 状态 | 说明 | 操作场景 |
|------|------|---------|
| 正常 | 可正常登录使用 | 默认状态 |
| 锁定 | 密码错误超限被锁定 | 安全策略自动触发 |
| 禁用 | 管理员手动禁用 | 员工离职、休假等 |
| 过期 | 账号超过有效期 | 临时账号自动过期 |

### 3.2 角色与RBAC配置

**预置角色说明：**

| 角色名称 | 权限范围 | 适用人员 |
|---------|---------|---------|
| 系统管理员 | 系统全部管理权限 | 运维管理员 |
| 安全分析师 | 告警分析、案件管理、威胁狩猎 | 安全分析师 |
| 审计员 | 只读权限，可查看审计日志 | 审计/合规人员 |
| 运营主管 | 告警、案件、报表管理 | 安全运营主管 |
| 只读用户 | 所有页面只读访问 | 管理层、客户 |
| 租户管理员 | 租户内全部管理权限 | MSSP客户管理员 |

**自定义角色创建：**

1. 进入「系统管理」→「角色管理」→「新增角色」
2. 输入角色名称和描述
3. 权限配置采用**模块+操作**的矩阵模式：
   - 模块：告警管理、案件管理、系统管理、数据接入等
   - 操作：查看、创建、编辑、删除、审核、导出
4. 保存后即可在创建用户时分配该角色

**权限验证：**

创建完成后，使用该角色的用户登录验证：
- 用户应只能看到角色授权的菜单和功能
- 未授权的操作按钮应置灰或隐藏或置灰
- API访问同样受权限控制，返回403

### 3.3 RBAC最佳实践

- **最小权限原则**：只赋予用户完成工作所需的最小权限
- **角色分离**：管理员权限与分析师权限分离，避免权限滥用
- **定期审计**：每月审查一次用户和角色分配情况
- **临时授权**：临时任务使用带过期时间的角色授权
- **双人复核**：敏感操作（如删除告警、修改策略）可启用审批流

## 模块4：多租户管理

### 4.1 租户模型

SecMind多租户架构支持两种模式：

| 模式 | 隔离级别 | 适用场景 |
|------|---------|---------|
| 数据隔离 | 租户数据逻辑隔离 | MSSP、集团多子公司 |
| 资源隔离 | 独立资源配额 | 高安全需求场景 |

每个租户拥有独立的：
- 告警数据空间（不可跨租户查看）
- 用户体系和角色配置
- 系统配置和通知策略
- 数据保留策略
- 资源使用配额

### 4.2 创建租户

**操作步骤：**

1. 以超级管理员身份登录，进入「多租户」→「租户管理」
2. 点击「创建租户」
3. 填写租户信息：
   - 租户名称：如"XX银行安全运营中心"
   - 租户编码：唯一标识，如"bank-xx"
   - 联系人信息：租户管理员姓名、邮箱、电话
   - 资源配额：设置告警处理量上限、存储空间上限
   - 有效期限：租户授权有效期
   - 数据保留策略：该租户的数据保留天数
4. 点击「保存」

**创建租户管理员：**

创建租户后，系统自动引导创建该租户的管理员账号：
- 输入管理员账号用于登录该租户的管理后台。

### 4.3 资源配额管理

**可配额资源：**

| 资源类型 | 配额项 | 超限处理 |
|---------|-------|---------|
| 告警处理 | 日处理告警上限 | 超出部分排队等待 |
| 存储空间 | 数据存储上限 | 达到80%发出预警 |
| 用户数 | 最大用户数 | 超出不可创建新用户 |
| AI调用 | 每日AI分析次数 | 超出后降级为规则分析 |
| API调用 | 每分钟API请求数 | 返回429限流 |

**配置方法：**

进入「多租户」→「资源配额」→选择租户→编辑各配额值→保存

### 4.4 数据隔离验证

**验证步骤：**

1. 使用租户A的管理员账号登录，创建一条测试告警
2. 使用租户B的管理员账号登录，确认无法看到租户A的告警
3. 使用API查询，确认请求头中的租户ID不同返回数据不同
4. 验证跨租户的数据导出被拦截

## 模块5：AI模型配置

### 5.1 AI模型架构

SecMind v2.3.0内置以下AI模型：

| 模型名称 | 版本 | 功能定位 | 硬件要求 |
|---------|------|---------|---------|
| secmind-llm-v2 | v2.3.0 | 安全大语言模型（核心推理） | NVIDIA A10/A100/昇腾310 |
| secmind-classifier-v3 | v2.3.0 | 告警分类与优先级排序 | CPU即可 |
| secmind-gnn-v2 | v2.3.0 | 攻击链路图分析 | CPU/GPU均可 |
| secmind-ti-enricher | v2.3.0 | 威胁情报富化 | CPU即可 |

### 5.2 模型管理与切换

**操作步骤：**

1. 进入「AI中心」→「模型管理」
2. 查看当前已部署的模型列表和版本
3. 点击要切换的模型，选择目标版本
4. 点击「部署」，系统自动拉取新版本镜像
5. 部署完成后，选择「设为默认」
6. 系统提示"模型切换成功，新模型将在5分钟内生效"

**模型回滚：**

如在切换后发现效果下降，可在「模型管理」→「版本历史」中选择上一个版本，点击「回滚」。

### 5.3 模型参数调优

**可调优参数：**

| 参数名称 | 默认值 | 范围 | 说明 |
|---------|-------|------|------|
| temperature | 0.3 | 0-1 | 推理随机性，越低越保守 |
| top_p | 0.9 | 0-1 | 采样阈值 |
| max_tokens | 2048 | 128-8192 | 最大输出长度 |
| confidence_threshold | 80 | 0-100 | 置信度阈值，低于此值需人工复核 |
| batch_size | 16 | 1-64 | 批处理大小 |
| max_retries | 3 | 0-10 | AI调用失败重试次数 |

**调优策略建议：**

```yaml
# 高性能模式（适合对告警量大的场景）
temperature: 0.2
batch_size: 32
confidence_threshold: 70

# 高精度模式（适合对准确性要求高的场景）
temperature: 0.1
batch_size: 8
confidence_threshold: 90

# 平衡模式（默认推荐）
temperature: 0.3
batch_size: 16
confidence_threshold: 80
```

### 5.4 监控推理质量

**关键监控指标：**

| 指标 | 说明 | 告警阈值 |
|------|------|---------|
| AI推理准确率 | 人工确认次数/总AI推理次数 | <85%触发告警 |
| AI推理覆盖率 | AI完成推理的告警占比 | <90%触发告警 |
| AI推理耗时 | 单条告警平均推理时间 | >30秒触发告警 |
| 模型调用成功率 | AI服务调用成功率 | <99%触发告警 |
| 置信度分布 | 各置信度区间告警分布 | 低置信度占比>20% |

**查看方法：**

进入「AI中心」→「模型监控」，可查看以上指标的实时图表和历史趋势。

## 模块6：系统监控与告警

### 6.1 服务器资源监控

**监控指标：**

| 指标 | 采集方式 | 告警阈值 |
|------|---------|---------|
| CPU | node_exporter | CPU使用率>80%持续5分钟 |
| 内存 | node_exporter | 内存使用率>85% |
| 磁盘 | node_exporter | 磁盘使用率>85%（数据盘）/ >90%（系统盘） |
| 网络 | node_exporter | 网络延迟>100ms |
| GPU | nvidia_exporter | GPU利用率>95% / 显存>90% |

**配置方法：**

1. 在所有节点上部署node_exporter
2. 在Prometheus配置中添加target
3. 在Grafana中导入系统监控Dashboard
4. 配置告警通知规则

### 6.2 服务健康检查

**服务状态查看：**

```bash
# 查看所有服务状态
secmind-cli status

# 查看单个服务详情
secmind-cli inspect secmind-engine

# 查看服务日志
secmind-cli logs secmind-api --tail 100
```

**关键服务健康检查点：**

| 服务 | 检查方法 | 正常状态 |
|------|---------|---------|
| PostgreSQL | `pg_isready` | accepting connections |
| Redis | `redis-cli ping` | PONG |
| Elasticsearch | `curl :9200/_cluster/health` | status: green/yellow |
| secmind-api | `curl :8080/api/health` | {"status": "ok"} |

**自动恢复机制：**

系统内置服务自愈能力：
- 服务崩溃：Docker/K8s自动重启，最多3次
- 健康检查失败：自动重启容器
- 数据库连接中断：自动重连，指数退避

### 6.3 资源预警配置

## 模块8：备份与恢复操作

### 8.1 备份策略规划

**备份分类：**

| 数据类型 | 备份方式 | 频率 | 保留期 |
|---------|---------|------|-------|
| 业务数据库（PostgreSQL） | pg_dump | 每日全量+每6小时增量 | 30天 |
| 告警索引（Elasticsearch） | snapshot | 每日 | 15天 |
| 配置文件 | rsync/tar | 每次变更后 | 90天 |
| 模型文件 | 目录打包 | 每次更新后 | 永久（保留版本） |

### 8.2 数据库备份

**pg_dump全量备份：**

```bash
#!/bin/bash
# backup_postgres_full.sh

BACKUP_DIR="/data/backup/postgres"
DATE=$(date +%Y%m%d_%H%M%S)
DB_NAME="secmind"
DB_USER="secmind"
DB_PASSWORD="your_password"

mkdir -p ${BACKUP_DIR}/${DATE}

# 执行全量备份
PGPASSWORD=${DB_PASSWORD} pg_dump -h localhost -U ${DB_USER} -d ${DB_NAME} \
  --format=custom \
  --compress=9 \
  --file=${BACKUP_DIR}/${DATE}/secmind_full.dump

# 备份数据库角色
PGPASSWORD=${DB_PASSWORD} pg_dumpall -h localhost -U ${DB_USER} --globals-only \
  --file=${BACKUP_DIR}/${DATE}/secmind_globals.sql

# 校验备份文件完整性
pg_restore -l ${BACKUP_DIR}/${DATE}/secmind_full.dump > /dev/null 2>&1
if [ $? -eq 0 ]; then
    echo "[成功] 备份文件完整性校验通过"
else
    echo "[失败] 备份文件损坏"
    exit 1
fi

# 清理超过30天的备份
find ${BACKUP_DIR} -maxdepth 1 -type d -mtime +30 -exec rm -rf {} \;

echo "[完成] 备份完成，文件大小：$(du -sh ${BACKUP_DIR}/${DATE}/secmind_full.dump)"
```

### 8.3 配置文件备份

```bash
#!/bin/bash
# backup_config.sh

BACKUP_DIR="/data/backup/config"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)

# 创建配置备份

  /opt/secmind/.env \
  /opt/secmind/config/*.yaml \

# 保留最近10份配置备份
ls -t ${BACKUP_DIR}/secmind-config-*.tar.gz | tail -n +11 | xargs -I {} rm {}

echo "[完成] 配置备份完成"
```

### 8.4 恢复演练

**数据库恢复步骤：**

```bash
#!/bin/bash
# restore_postgres.sh

RESTORE_FILE=$1

if [ -z "$RESTORE_FILE" ]; then
    echo "用法：./restore_postgres.sh <备份文件路径>"
    exit 1
fi

# 停止应用服务（恢复前必须停止）
systemctl stop secmind-api secmind-worker

# 创建新数据库
createdb -U secmind secmind_restore

# 恢复数据

# 验证恢复

psql -U secmind -d secmind_restore -c "SELECT count(*) FROM cases;"

# 切换数据库连接
# 修改.env中的DB_NAME=secmind_restore

# 重启应用服务
systemctl start secmind-api secmind-worker

echo "[完成] 恢复完成，请验证数据完整性"
```

## 模块9：故障排查指南

### 9.1 故障排查流程图

```
服务不可用
    │
    ├─→ 检查服务状态（systemctl status/ docker ps）
    │      │
    │      ├─→ 服务未运行 → 启动服务 / 排查崩溃原因
    │      │                    │
    │      │                    └─→ 查看日志（journals -u / docker logs）
    │      │
    │      └─→ 服务正常运行 → 检查端口监听（netstat -tlnp）
    │                              │
    │                              └─→ 端口未监听 → 检查配置 / 网络
    │
    └─→ Web无法访问
           │
           ├─→ Nginx状态检查 → Nginx异常 → 检查Nginx配置和日志
           │
           ├─→ API无法响应 → 检查API日志 → 数据库连接是否正常
           │
           └─→ 数据库故障 → 检查PostgreSQL状态
                                  │
                                  └─→ 主从同步异常 → 重建slave
```

### 10.2 升级执行

**标准升级流程：**

```bash
# 步骤1：获取新版本安装包
wget https://download.secmind.com/releases/secmind-v2.3.0.tar.gz
tar -zxvf secmind-v2.3.0.tar.gz
cd secmind-v2.3.0

# 步骤2：执行升级前检查
./upgrade.sh --check-only

# 步骤3：停止服务（停止非核心服务，保留数据库）
docker-compose stop secmind-api secmind-worker secmind-engine
docker-compose stop secmind-scheduler secmind-collector

# 步骤4：备份现有配置
cp /opt/secmind/.env /opt/secmind/.env.backup.$(date +%Y%m%d)

# 步骤5：导入新版本镜像
docker load -i images/secmind-api-v2.3.0.tar
docker load -i images/secmind-engine-v2.3.0.tar
# ... 导入所有新版本镜像

# 步骤6：执行数据库迁移
./upgrade.sh --migrate-db

# 步骤7：更新配置文件
./upgrade.sh --update-config

# 步骤8：启动所有服务
docker-compose up -d

# 步骤9：验证升级
./upgrade.sh --verify
```

### 10.3 升级后验证

**验证清单：**

1. **服务检查**：所有服务运行正常，版本号显示为v2.3.0
2. **功能检查**：Web界面正常，各功能模块可正常使用
3. **AI检查**：AI推理正常，模型版本已更新
4. **数据检查**：告警数据完整，历史数据可正常查询

6. **数据源检查**：所有数据源连接正常
7. **通知检查**：发送测试通知确认通知渠道正常

### 10.4 回滚操作

**回滚条件：**

- 升级后验证失败且无法修复
- 升级后出现严重影响业务的故障
- 关键功能不兼容
