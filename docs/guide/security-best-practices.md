# 安全最佳实践

## 概述

### 安全原则

SecMind 遵循以下核心安全原则指导架构设计和开发实践：

- **纵深防御**：在网络层、应用层、数据层分别设置安全控制措施，单一防线失效不影响整体安全
- **最小权限**：所有用户和服务只授予完成本职工作所需的最小权限集
- **默认安全**：所有安全特性默认启用，无需用户额外配置即可获得基础安全防护
- **数据保护优先**：数据在传输中和存储中均进行加密，敏感字段在应用层额外加密
- **可审计性**：所有敏感操作记录审计日志，确保安全事件可追溯、可还原

### 威胁模型

SecMind 作为安全运营平台，主要面临以下威胁：

| 威胁类别 | 描述 | 缓解措施 |
|---------|------|---------|
| 未授权访问 | 攻击者通过弱口令、Token 泄露等方式获取系统访问权限 | JWT 认证、密码策略、登录锁定 |
| API 滥用 | 恶意调用 API 导致服务不可用或数据泄露 | 速率限制、请求验证、CORS 策略 |
| 数据泄露 | 多租户场景下租户数据被其他租户访问 | RLS 隔离、RBAC 权限控制 |
| 注入攻击 | SQL / NoSQL / 命令注入导致数据泄露或系统控制 | ORM 参数化查询、输入验证 |
| 供应链攻击 | 第三方依赖存在已知漏洞被利用 | 依赖扫描、定期更新、SBOM 管理 |

## 认证安全

### JWT 配置

JWT（JSON Web Token）是 SecMind API 认证的核心机制，请遵循以下配置规范：

- **使用强密钥**：密钥长度至少 256 位（32 字节），推荐使用 `openssl rand -hex 32` 生成。禁止使用硬编码或弱密钥
- **设置合理的过期时间**：Access Token 过期时间建议设置为 30 分钟，最长不超过 2 小时。过期时间过长会增加 Token 泄露风险
- **启用 Token 刷新机制**：使用 Refresh Token 实现无感续期，Refresh Token 过期时间建议为 7 天。刷新接口需要额外的安全校验（如客户端指纹）
- **定期轮换密钥**：建议每 90 天轮换一次 JWT 签名密钥。密钥轮换时应保留前一个密钥的验证能力（双密钥模式），避免已签发的 Token 立即失效
- **Token 存储**：前端禁止将 Token 存储在 `localStorage` 中，应使用 `httpOnly` Cookie 或内存变量存储。移动端应使用系统安全存储区

### 密码策略

- **最小长度**：密码最小长度为 8 位，建议 12 位以上
- **复杂度要求**：密码必须同时包含以下四类字符中的至少三类：
  - 大写字母（A-Z）
  - 小写字母（a-z）
  - 数字（0-9）
  - 特殊字符（`!@#$%^&*()_+-=[]{}|;':\",./<>?`）
- **哈希存储**：所有密码使用 bcrypt 算法进行哈希存储，cost factor 设置为 12。禁止使用 MD5、SHA-1、SHA-256 等快速哈希算法存储密码
- **登录失败锁定**：连续 5 次登录失败后，账户锁定 15 分钟。锁定期间即使输入正确密码也无法登录。锁定状态通过 Redis 缓存实现，服务重启后锁定状态不丢失
- **密码历史**：禁止使用最近 5 次使用过的密码。修改密码时需要验证旧密码
- **初始密码**：首次创建用户或重置密码时，系统生成随机密码并要求用户在首次登录时修改

## 传输安全

- **强制 HTTPS**：所有 API 通信必须使用 HTTPS 协议。生产环境禁止使用 HTTP。开发环境可使用自签名证书，但不得用于生产
- **TLS 1.2+**：仅支持 TLS 1.2 和 TLS 1.3 协议。禁用 SSL 3.0、TLS 1.0 和 TLS 1.1。密码套件优先使用 ECDHE + AES-GCM 组合
- **HSTS 头**：所有 HTTP 响应必须包含 `Strict-Transport-Security` 头，`max-age` 至少为 31536000（1 年），建议包含 `includeSubDomains` 和 `preload`
- **证书自动续期**：使用 Let's Encrypt 或类似服务实现证书自动续期。证书到期前 30 天开始自动检查并续期。续期失败时发送告警通知

## API 安全

- **速率限制**：按 IP 地址和用户 ID 双重维度进行速率限制。默认限制为每 IP 每分钟 60 次请求。敏感接口（登录、注册、密码重置）限制为每 IP 每分钟 10 次。超出限制返回 `429 Too Many Requests`
- **请求体大小限制**：API 请求体大小限制为 10MB。文件上传接口单独配置上限（100MB）。超出限制返回 `413 Payload Too Large`
- **CORS 白名单**：严格配置跨域访问白名单，仅允许受信任的前端域名。禁止使用 `Access-Control-Allow-Origin: *`。CORS 预检请求缓存时间设置为 1 小时
- **SQL 注入防护**：所有数据库查询必须使用 ORM 参数化查询或预处理语句，禁止拼接 SQL 字符串。动态表名和字段名必须经过白名单校验
- **XSS 防护**：所有用户输入在输出到前端时进行 HTML 实体编码。富文本内容使用安全的 Markdown 渲染库，禁止直接渲染原始 HTML。设置 `Content-Security-Policy` 头限制脚本执行
- **请求验证**：所有 API 请求参数进行严格的类型校验和边界检查。枚举类型参数进行白名单校验。数值类型参数检查范围边界

## 数据安全

### 数据加密

| 阶段 | 加密方式 | 说明 |
|------|---------|------|
| 传输中 | TLS 1.2+ | 所有网络通信使用 TLS 加密，确保数据在传输过程中不被窃听或篡改 |
| 存储中 | 数据库透明加密（TDE） | PostgreSQL 数据文件使用 AES-256 加密，防止物理存储介质泄露导致数据泄露 |
| 敏感字段 | 应用层 AES-256-GCM | API Key、密钥等敏感字段在入库前使用应用层加密，加密密钥与数据分离存储 |

敏感字段加密规范：
- 使用 AES-256-GCM 认证加密模式，提供机密性和完整性保护
- 每个敏感字段使用独立的初始化向量（IV）
- 加密密钥通过密钥管理服务（KMS）或硬件安全模块（HSM）管理
- 禁止将加密密钥与加密数据存储在同一数据库或同一服务器中

### 数据隔离

- **多租户 RLS**：使用 PostgreSQL Row-Level Security 实现租户级别的数据隔离。所有数据表通过租户 ID 字段自动过滤，应用层无需额外处理。RLS 策略由数据库管理员统一管理，应用用户无法绕过
- **RBAC 最小权限**：基于角色的访问控制系统，每个角色只授予完成工作所需的最小权限。角色权限矩阵在管理后台集中配置。禁止直接为用户分配权限，必须通过角色间接授权
- **审计日志**：所有数据访问和修改操作记录审计日志，包括：操作时间、操作用户、操作类型、操作对象、操作前后数据摘要。审计日志独立存储，禁止非授权修改和删除。审计日志保留时间不少于 180 天

## 运维安全

- **定期安全更新**：每月执行一次安全更新，包括操作系统补丁、数据库补丁、运行时环境更新。紧急安全漏洞（CVSS 7.0+）在 48 小时内完成修复
- **漏洞扫描**：每周执行一次自动化漏洞扫描，覆盖 Web 应用、API、依赖库。每次版本发布前执行完整安全扫描。扫描结果记录并跟踪修复进度
- **入侵检测**：部署 WAF（Web 应用防火墙）防护常见 Web 攻击。配置入侵检测系统（IDS）监控异常流量。设置安全告警规则，异常行为实时告警
- **日志监控**：集中收集和存储系统日志、应用日志、安全日志。日志格式标准化，支持结构化查询。设置日志告警规则，关键安全事件实时通知
- **备份策略**：数据库每日全量备份，保留 30 天。配置文件每次变更前自动备份。备份数据加密存储，与生产环境物理隔离。每月执行一次备份恢复演练，验证备份可用性
- **容器安全**：基础镜像使用最小化操作系统，定期扫描镜像漏洞。容器运行时禁止以 root 用户运行。使用只读文件系统，避免容器内写操作

## 合规要求

### 等保 2.0 合规检查项

SecMind 在架构设计中遵循等保 2.0 三级安全要求，以下为关键检查项：

| 控制点 | 要求 | SecMind 实现 |
|--------|------|-------------|
| 身份鉴别 | 双因素认证、登录失败锁定 | JWT + 密码策略 + 登录锁定 |
| 访问控制 | 最小权限、角色管理 | RBAC + RLS 数据隔离 |
| 安全审计 | 审计日志覆盖所有敏感操作 | 完整审计日志记录 |
| 入侵防范 | Web 攻击防护、恶意代码防范 | WAF + 输入验证 + 参数化查询 |
| 数据完整性 | 传输和存储完整性保护 | TLS + AES-256-GCM |
| 数据保密性 | 数据传输和存储加密 | 传输加密 + 存储加密 + 字段加密 |

### GDPR 数据处理记录

- **数据分类**：明确区分个人数据（PII）和非个人数据。PII 包括用户名、邮箱、IP 地址、登录日志等
- **数据处理记录**：所有涉及 PII 的数据处理操作记录在案，包括处理目的、数据类型、存储位置、保留期限
- **数据主体权利**：支持数据主体行使访问权、更正权、删除权（被遗忘权）、限制处理权、数据可携带权
- **数据泄露通知**：发生数据泄露事件后 72 小时内通知监管机构和受影响的数据主体
- **数据处理协议**：与第三方数据处理者（如 LLM API 供应商）签订数据处理协议（DPA），确保符合 GDPR 要求

### 数据保留策略

| 数据类型 | 保留期限 | 说明 |
|---------|---------|------|
| 用户账号数据 | 账户存续期间 | 账户删除后 30 天内彻底清除 |
| 安全告警数据 | 365 天 | 超过期限的数据自动归档到冷存储 |
| 审计日志 | 180 天 | 满足等保 2.0 合规要求 |
| IOC 数据 | 按配置策略 | 支持自定义保留期限，默认 90 天 |
| 会话 Token | 至过期 | Access Token 30 分钟，Refresh Token 7 天 |
| 操作缓存 | 按缓存策略 | Redis 缓存根据 LRU 策略自动淘汰 |
