# SecMind 后台 UI 统一规范 v1

版本：v1.0  
适用范围：`frontend/src/app/(dashboard)` 全部后台页面  
目标：统一视觉语言、提升 B 端精致感、满足上线标准  
当前基准页：`/investigate` AI 研判页

---

## 1. 规范目标

这份规范解决三个问题：

1. 全站后台页面必须说同一种设计语言。
2. 页面要体现 B 端产品应有的专业、可信、克制、耐看。
3. 设计输出必须达到上线可交付标准，而不是“开发中可用状态”。

这份规范优先级高于历史的深色赛博风描述。  
当前后台统一采用：

- 浅色工作台风格
- 高可读性优先
- 低噪音、高秩序
- AI 模块允许有识别度，但不能独立成另一套 UI

---

## 2. 设计总原则

### 2.1 核心原则

1. 先可读，再精致。
2. 先结构清晰，再做视觉强调。
3. 先统一组件，再做页面个性。
4. 颜色服务于状态，不服务于装饰。
5. 页面必须表达业务逻辑，而不是只摆信息块。

### 2.2 B 端审美原则

后台“好看”的标准不是炫，而是：

- 信息主次清楚
- 操作路径清楚
- 状态表达稳定
- 阅读不费力
- 长时间使用不疲劳

禁止为了“科技感”牺牲：

- 对比度
- 信息密度控制
- 表格可读性
- 状态一致性
- 页面逻辑完整性

---

## 3. 视觉语言

### 3.1 整体气质

SecMind 后台应传达：

- 专业
- 冷静
- 可控
- 高效
- 可信

关键词：

- `Structured`
- `Precise`
- `Low-noise`
- `Operational`

禁止：

- 赛博霓虹泛滥
- 大面积 glow
- 大量透明叠层
- 过多高饱和状态色
- 一页同时出现多种“主视觉主角”

---

## 4. 色彩系统

### 4.1 基础中性色

后台统一使用 Slate 系为基础灰阶。

- 页面主背景：`#f8fafc` ~ `#f1f5f9`
- 主卡片背景：`#ffffff`
- 次级浅底：`#f8fafc`
- 主边框：`#e2e8f0`
- 弱边框：`#edf2f7`

### 4.2 文本颜色

- 一级标题 / 核心数字：`text-slate-950` 或 `text-slate-900`
- 二级标题 / 重要标签：`text-slate-800` / `text-slate-700`
- 正文：`text-slate-600`
- 辅助说明 / 时间 / Meta：`text-slate-500`
- 禁止在浅底正文中使用：`text-slate-400`

### 4.3 品牌强调色

品牌主色只保留一个核心方向：

- 主交互 / AI / 高亮筛选：`cyan`

推荐：

- 主交互文字：`text-cyan-700`
- 主交互浅底：`bg-cyan-50`
- 主交互边框：`border-cyan-200`
- 主按钮：`bg-cyan-600 hover:bg-cyan-700 text-white`

禁止：

- 在白底内容区大面积使用 `text-cyan-300/400`
- 用 cyan 作为正文颜色

### 4.4 状态色

状态色必须跨页面一致：

- 严重 / 高风险：红
- 警示 / 待处理：橙 / 黄
- 处理中 / AI / 分析中：青 / 蓝
- 成功 / 已闭环 / 已确认：绿
- 中性 / 停用 / 已静默 / 已归档：灰

### 4.5 紫色使用规则

紫色不再作为通用主状态色。  
仅允许在以下场景使用：

- MITRE / 攻击技术映射
- 辅助分析模块
- 非主路径的 AI 语义信息

禁止：

- 把紫色作为页面主交互色
- 用紫色表达“处理中”
- 一页里同时把 cyan 和 purple 都当主强调色

---

## 5. 页面结构规范

后台页面统一使用 4 层结构：

1. `Page Header`
2. `Summary / Filters`
3. `Primary Content`
4. `Secondary Content / Detail`

### 5.1 Page Header

用于承载：

- 页面名称
- 页面副标题
- 页面级动作

禁止在 Header 中塞入：

- 复杂筛选器
- 大量统计卡
- 页面内二级状态

Header 只解决“这是哪一页”和“这里的顶级动作是什么”。

### 5.2 Summary 区

用于承载：

- 全局统计
- 当前工作状态
- 高层摘要

Summary 区必须回答：

- 当前总量是多少
- 当前最需要关注什么
- 这些数字之间是什么关系

禁止出现：

- 互相不能解释的数字
- 数字总量与状态拆分不闭环
- 同一信息在左右两个模块重复出现

### 5.3 Filters 区

用于承载：

- 搜索
- 状态筛选
- 时间范围
- 来源筛选

Filters 区只放操作，不放概念包装。  
不要起“工作台控制台”这种无法直接表达业务用途的名字。

推荐命名：

- 筛选与状态
- 筛选条件
- 当前视图
- 工作区筛选

### 5.4 Primary Content

页面主内容只能有一个主角：

- 表格
- 案件池
- 工作流画布
- 证据链详情
- 统计图

必须明确：

- 用户进入页面先看哪里
- 主内容的默认态是什么
- 主内容选中之后会发生什么

### 5.5 Secondary Content

用于承载：

- 详情
- 侧边摘要
- 阅读提示
- 辅助数据

禁止出现“大片空白但没有明确用途”的区域。  
如果右侧存在大面积空间，必须有明确角色：

- 详情预览
- 辅读摘要
- 状态提示
- 操作建议

---

## 6. 卡片系统

后台卡片只保留 3 类。

### 6.1 Primary Card

用途：

- 页面主容器
- 核心详情
- 表格容器
- 工作台主面板

样式：

- `bg-white`
- `border-slate-200`
- 轻阴影
- 圆角 16~24

### 6.2 Soft Card

用途：

- 筛选区
- 次级信息区
- 轻量统计块
- 页面辅助摘要

样式：

- `bg-white` 或 `bg-slate-50`
- `border-slate-200`
- 阴影更轻

### 6.3 State Card

用途：

- 风险状态
- 待复核
- AI 建议
- 成功闭环

规则：

- 只能轻量染色
- 底色透明度低
- 边框明确
- 文字对比清楚

禁止：

- 状态卡大面积发光
- 状态卡透明叠透明
- 状态卡抢主内容风头

---

## 7. 统计卡规范

统计卡必须满足：

1. 数字有解释
2. 统计之间有关系
3. 同一页数字风格一致

### 7.1 统计卡结构

统一结构：

- Label
- Value
- Optional Hint

可选：

- 图标
- 环比 / 状态说明

### 7.2 数字闭环要求

如果页面展示：

- 全部案件
- 研判中
- 待复核
- 处置中
- 已闭环

则必须满足：

- `全部 = 各状态求和`

并建议在 `全部` 卡片内直接给出解释提示。

禁止：

- 数字独立摆放但互相解释不清
- “总数”与“状态数”来自不同口径

### 7.3 数字排版

- 核心数字：`text-xl` ~ `text-3xl`
- 使用 `tabular` 感较强的排布
- 不允许图标、数字、说明出现挤压换行

---

## 8. 表单与筛选器规范

### 8.1 输入框

统一要求：

- 高度固定
- 文本对比足够
- 白底或极浅底
- Focus 边框统一

推荐：

- `border-slate-200`
- `bg-white`
- `text-slate-900`
- `placeholder:text-slate-400`
- `focus:border-cyan-400/60`

### 8.2 搜索 + 筛选组合

推荐顺序：

1. 搜索
2. 状态筛选
3. 来源 / 时间 / 类型
4. 清除筛选

禁止：

- 把搜索框做成主视觉
- 筛选器视觉比主内容更强
- 过多同权重筛选器平铺

---

## 9. 表格规范

### 9.1 表格头

- 统一使用弱灰文字
- 统一行高
- 统一左对齐逻辑

### 9.2 表格行

- 行 hover 轻微
- 不可出现高饱和整行染色
- 高风险行可以极轻提示底色，但不能影响阅读

### 9.3 表格内标签

- 标签是辅助，不应比正文更强
- 表格中的 badge 要比卡片中的 badge 更克制

---

## 10. Badge / 状态标签规范

状态 Badge 统一要求：

- 高度统一
- 文字字号统一
- 内边距统一
- 边框明确

### 10.1 Badge 分类

1. `Status Badge`
- 业务状态：已确认、处理中、已闭环

2. `Type Badge`
- 类型：EDR、告警级别、资产类型、MITRE 技术

3. `Tag Badge`
- 普通标签、实体名、附加说明

### 10.2 使用原则

- 状态 Badge 强于 Type Badge
- Type Badge 强于普通 Tag

禁止：

- 同一块区域同时出现太多彩色 Badge
- 用 badge 代替正文

---

## 11. 按钮规范

### 11.1 层级

后台按钮只保留三层：

1. Primary
- 主动作
- `bg-cyan-600 text-white`

2. Secondary
- 次动作
- 白底 + 中性色或青色文字

3. Tertiary / Ghost
- 行内辅助动作
- 轻 hover

### 11.2 禁止

- 一个区域同时出现多个 Primary
- 同一行按钮颜色语义混乱
- 重要主按钮用浅色镂空方式表达

---

## 12. 空状态规范

空状态必须包含：

1. 图标
2. 主句
3. 解释句
4. 下一步建议（可选）

空状态不能只是“暂无数据”。  
必须说明：

- 为什么为空
- 用户下一步可以做什么

---

## 13. Dialog 规范

### 13.1 结构

- 标题
- 说明
- 表单 / 内容
- Footer

### 13.2 Footer

统一规则：

- 左侧留空或辅助说明
- 右侧为取消 + 确认
- 主按钮永远在最右侧

禁止：

- Footer 样式每页都不同
- 确认按钮和取消按钮视觉权重混乱

---

## 14. AI 模块专属规范

AI 模块必须有识别度，但不能成为另一套主题。

### 14.1 可用表达

- 轻 cyan 强调
- 专属标题图标
- AI 摘要块
- 推理链模块
- 步骤导航

### 14.2 禁止表达

- 大面积未来科技感渐变
- 强 glow
- 赛博霓虹卡片
- 把 AI 模块做成暗色孤岛

### 14.3 AI 页面规则

AI 页面必须满足：

1. 用户一眼知道当前在看什么问题
2. AI 的结论和证据必须分层
3. 用户不能一次被全部推理链压垮
4. 阅读路径必须可控

推荐阅读结构：

- 总览
- 当前结论
- 当前建议
- 分步证据链
- 闭环反馈

---

## 15. AI 研判页样板规则

`/investigate` 是当前后台的样板页。

### 15.1 这个页面应承担的职责

1. 作为 AI 工作台样板
2. 作为“复杂信息阅读页”样板
3. 作为“案件池 + 详情预览 + 深入查看”样板

### 15.2 强制规则

1. 顶部总览只讲总体，不重复筛选逻辑
2. 筛选区只讲操作，不造新概念词
3. 左侧案件池和右侧预览必须形成联动
4. 未选中状态不能出现大块无意义空白
5. 完整证据链不能一次性全部倾倒给用户

### 15.3 证据链阅读规则

推荐：

- 步骤导航
- 单步聚焦
- 阅读提示
- 左右分栏

不推荐：

- 一个页面从上到下直接堆全部步骤
- 一个步骤里又堆太多不同类型卡片而无优先级

如果链路很长，优先考虑：

1. 默认只显示关键步骤
2. 展开查看全部
3. 单步聚焦查看

而不是直接弹窗化。  
弹窗只适合短详情，不适合主工作流阅读。

---

## 16. 命名规范

页面内区块命名必须业务清晰。

### 16.1 推荐命名

- 总览
- 筛选与状态
- 案件池
- 详情预览
- 完整证据链
- AI 建议
- 闭环结论

### 16.2 禁止命名

- 工作台控制台
- 智能引擎中枢
- 协同决策矩阵
- 其他只听起来高级但用户不知道是什么意思的词

原则：

名字必须直接告诉用户“这里是什么”。

---

## 17. 动效规范

动效只能辅助状态，不可主导页面。

允许：

- 轻 hover 阴影
- 轻位移
- loading
- pulse 点状状态提示

禁止：

- 大面积 glow 动画
- 高频闪动
- 多模块同时做运动

---

## 18. 上线验收标准

上线前每个后台页面必须通过以下检查：

### 18.1 视觉一致性

- 是否符合统一页面结构
- 卡片是否只用 3 类之一
- 状态色是否符合全站规则
- 搜索 / 筛选器高度是否统一
- 空状态是否统一

### 18.2 业务逻辑一致性

- 数字是否闭环
- 状态是否可解释
- 板块关系是否清楚
- 是否存在无意义空白区域

### 18.3 可读性

- 主次是否明确
- 长内容是否可分步阅读
- 表格是否清楚
- Badge 是否过多

### 18.4 交互完成度

- 默认态是否合理
- 加载态是否存在
- 空态是否存在
- 错误态是否存在
- 成功态是否存在

### 18.5 细节完整度

- 无明显换行挤压
- 无孤立无解释数字
- 无奇怪命名
- 无历史风格残留块

---

## 19. 当前版本的禁用项

从现在开始，后台页面禁止继续新增以下风格：

1. `bg-white/[0.02]` / `bg-white/[0.04]` 作为大面积主卡底色
2. 大面积 `backdrop-blur-xl`
3. 无业务理由的 `shadow-[0_0_xx] glow`
4. 在白底正文上使用浅 cyan 文本
5. 紫色作为主交互色
6. 同页使用多种主风格卡片
7. 没解释关系的统计数字
8. 大块空白仅为了“留设计感”

---

## 20. 推荐执行顺序

如果从当前项目继续推进，建议按以下顺序执行：

1. 以 `/investigate` 定稿为样板页
2. 统一 `dashboard / notifications / response / signals`
3. 统一 `assets / hunting / audit / metrics / workflows`
4. 最后统一 `dialogs / badge / table / empty states`

---

## 21. 最终目标

SecMind 后台最终应达到：

- 第一眼像成熟 B 端产品
- 第二眼能看出 AI 能力是核心差异点
- 长时间使用不累
- 页面之间像一个系统，而不是一组页面

这份规范是后续全部页面设计与实现的唯一后台标准基线。
