## 第一章 产品简介与快速开始

### 1.1 产品概述

SecMind 是一款 **AI 驱动的智能安全运营平台**，面向企业安全运营中心（SOC）团队，提供从威胁检测、AI 智能研判、事件调查、自动处置到反馈学习的全链路安全运营能力。v3.0 版本引入 AI 可解释性推理链可视化、多模型智能路由、策略自演化、多租户管理等重大能力升级。

平台核心价值主张：

| 维度 | 能力描述 |
|------|----------|
| **AI 原生研判** | 基于 5 阶段推理链（事件发现→证据关联→模式匹配→置信度评估→处置决策），自动完成攻击意图判定 |
| **全生命周期覆盖** | 从原始信号采集到最终处置闭环，覆盖安全事件的完整生命周期 |
| **人机协同** | AI 自动处置 + 人工审批复核，兼顾效率与安全 |
| **ATT&CK 对齐** | 威胁狩猎模块基于 MITRE ATT&CK 框架，支持战术/技术级假设验证 |
| **持续进化** | 通过策略自演化机制，AI 研判准确率随运营数据积累持续提升 |
| **多模型路由** | 根据任务类型智能选择最优 LLM 模型，兼顾性能与成本 |
| **多租户隔离** | 原生支持多租户数据隔离（RLS），满足 MSP/MSSP 运营场景 |

平台共包含 **32 个功能页面**，分为公开页面（4 个）、认证页面（4 个）、后台功能页面（24 个）三大类。

### 1.2 技术架构

```
┌─────────────────────────────────────────────┐
│              前端 (Next.js 15)                │
│   React 19 + TypeScript + Tailwind CSS       │
│   ECharts 可视化 / shadcn/ui 组件库           │
│   WebSocket 实时通信                          │
├─────────────────────────────────────────────┤
│              后端 (Python FastAPI)            │
│   SQLAlchemy ORM / PostgreSQL 16             │
│   Redis 缓存 / Celery 异步任务                │
│   JWT 认证 / bcrypt 密码哈希                  │
│   WebSocket 服务 (SignalR 协议)               │
├─────────────────────────────────────────────┤
│              AI 层 (AI Layer)                 │
│   LLM Provider (OpenAI / Claude / 本地模型)   │
│   Model Router (智能模型路由)                  │
│   Prompt Templates (提示词模板管理)            │
│   推理链引擎 / 策略演化引擎 (Q-Learning)       │
├─────────────────────────────────────────────┤
│              数据层                           │
│   devices(数据源) / users(用户)               │
│   alerts(告警) / cases(案件) / audit_logs     │
│   tenants(租户) / billing(账单)               │
│   i18n 多语言 (zh/en)                         │
└─────────────────────────────────────────────┘
```

**技术栈详情：**

- **前端框架**：Next.js 15 (App Router) + React 19
- **UI 组件库**：shadcn/ui + Lucide Icons
- **状态管理**：Zustand (locale-store, user-store)
- **图表引擎**：ECharts (echarts-for-react)
- **后端框架**：Python FastAPI
- **数据库**：PostgreSQL 16（生产）/ SQLite（开发）
- **缓存与队列**：Redis + Celery 异步任务队列
- **AI 模型**：OpenAI GPT-4o / Claude Sonnet / 本地开源模型
- **模型路由**：智能 Model Router（按任务类型自动选择最优模型）
- **实时通信**：WebSocket（SignalR 协议）
- **容器化**：Docker + Docker Compose
- **认证方式**：JWT Token + SSO 回调支持
- **国际化**：内置中文/英文双语支持

### 1.3 安装部署

#### 1.3.1 环境要求

| 组件 | 最低要求 | 推荐配置 |
|------|----------|----------|
| 操作系统 | Linux/macOS/Windows | Ubuntu 22.04 LTS |
| Node.js | ≥ 18.x | ≥ 20.x LTS |
| Python | ≥ 3.10 | ≥ 3.12 |
| Docker | 24.x | 最新版（推荐容器化部署） |
| PostgreSQL | 14.x | 16.x |
| Redis | 6.x | 7.x |
| 内存 | 8 GB | 16 GB+ |
| 磁盘空间 | 20 GB | 100 GB+ SSD |
| 浏览器 | Chrome 90+ / Firefox 88+ / Edge 90+ | 最新版 |

#### 1.3.2 快速部署步骤

```bash
# 方式一：Docker Compose 部署（推荐）
docker compose up -d
# 访问 http://localhost:3000

# 方式二：手动部署
# 1. 克隆项目
git clone <repository-url>
cd SecMind

# 2. 启动后端服务
cd backend
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload

# 3. 启动前端开发服务（新终端）
cd frontend
npm install
npm run dev

# 4. 访问平台
# 前端: http://localhost:3000
# 后端 API: http://localhost:8000
# API 文档: http://localhost:8000/docs
```

#### 1.3.3 生产环境部署

```bash
# Docker Compose 生产部署
docker compose -f docker-compose.prod.yml up -d

# 或手动部署
# 前端构建
cd frontend
npm run build
npm start  # 生产模式启动于端口 3000

# 后端生产启动（建议使用 Gunicorn + Uvicorn）
cd backend
pip install gunicorn
gunicorn app.main:app -w 4 -k uvicorn.workers.UvicornWorker --bind 0.0.0.0:8000
```

> **注意**：生产环境请务必修改 `backend/.env` 中的以下配置项：
> - `SECRET_KEY`：JWT 签名密钥（必须为高强度随机字符串）
> - `DATABASE_URL`：切换至 PostgreSQL 16 连接串
> - `REDIS_URL`：Redis 连接地址
> - `CORS_ORIGINS`：限制为实际前端域名
> - `LLM_API_KEY`：AI 模型 API Key 配置

### 1.4 首次登录与引导流程

#### 1.4.1 注册账号

1. 访问平台首页 `http://your-domain`，点击右上角 **「注册」** 按钮
2. 进入注册页面 `/register`，填写以下信息：

| 字段 | 说明 | 要求 |
|------|------|------|
| 用户名 | 登录标识 | 3-20 字符，字母数字下划线 |
| 邮箱 | 用于通知和找回密码 | 合法邮箱格式 |
| 密码 | 登录凭证 | ≥8 位，含大小写字母和数字 |
| 确认密码 | 二次确认 | 与密码一致 |

3. 点击 **「注册」** 按钮，系统自动创建账号并跳转至登录页

#### 1.4.2 登录系统

1. 访问 `/login` 页面
2. 输入用户名（或邮箱）和密码
3. 点击 **「登录」** 或按 `Enter` 键
4. 支持 SSO 单点登录：通过 `/auth/callback` 回调完成第三方认证

#### 1.4.3 新手引导流程（Onboarding Guide）

首次成功登录后，系统会自动弹出 **5 步引导向导**，帮助新用户快速熟悉平台核心功能：

| 步骤 | 标题 | 内容要点 |
|------|------|----------|
| **Step 1** | 欢迎 | 平台介绍："SecMind 将 AI 能力注入安全运营的每个环节" |
| **Step 2** | AI 分析工作台 | 核心入口介绍：AI 研判界面是平台的核心，所有告警在此完成智能分析 |
| **Step 3** | 信号 → 告警 → 处置 | 生命周期说明：原始信号经 AI 筛选升级为告警，再触发自动或人工处置 |
| **Step 4** | 反馈驱动进化 | 学习机制说明：每次人工反馈都会让 AI 判断更精准 |
| **Step 5** | 开始使用 | 引导结束提示：可随时通过右上角菜单重新打开引导 |

**操作要点：**
- 每步引导卡片位于屏幕中央，带有半透明背景遮罩高亮目标区域
- 底部提供 **「上一步」** / **「下一步」** / **「跳过」** 三个操作按钮
- 最后一步提供 **「开始使用」** 主按钮，点击后关闭引导并进入仪表盘
- 完成引导后，引导状态会被持久化存储（localStorage），后续登录不再自动弹出
- 如需重新查看引导，点击右上角头像旁的 **「帮助」** 图标即可

## 第二章 界面导航说明

### 2.1 整体布局

后台页面采用经典的 **左右分栏布局**：

```
┌──────────┬──────────────────────────────────────┐
│          │  顶部导航栏 (Header)                   │
│          │  [Logo] [面包屑] [搜索] [通知] [头像]   │
│  侧边栏   ├──────────────────────────────────────┤
│ Sidebar  │                                      │
│          │         主内容区 (Main Content)        │
│ [5大分组] │                                      │
│ [20个菜单]│                                      │
│          │                                      │
│          │                                      │
└──────────┴──────────────────────────────────────┘
```

**布局区域说明：**

| 区域 | 位置 | 功能 |
|------|------|------|
| **顶部导航栏** | 页面最上方固定 | 包含 Logo、面包屑导航、全局搜索、通知铃铛、用户头像/语言切换 |
| **侧边栏** | 左侧固定宽度（可折叠） | 7 大功能分组、24 个功能菜单入口 |
| **主内容区** | 右侧自适应区域 | 当前选中功能的页面内容 |

**顶部导航栏详细元素（从左到右）：**

1. **SecMind Logo** — 点击返回仪表盘首页
2. **面包屑导航** — 显示当前页面路径（如：运营总览 > AI分析工作台）
3. **全局搜索框** — 支持模糊搜索告警、案件、IOC 等
4. **通知铃铛** — 显示未读通知数量红点，点击展开通知下拉面板
5. **语言切换** — 中文 / English 切换按钮
6. **用户头像** — 下拉菜单包含：个人资料、偏好设置、退出登录

## 第三章 核心操作手册（典型工作流）

本章通过四个典型工作流，串联平台各模块的使用方法，帮助用户建立完整的操作心智模型。

### 3.1 工作流一：从告警到处置的完整流程

**场景**：EDR 检测到一台终端出现异常行为，需要完成从信号确认到处置闭环的全过程。

#### 步骤 1：在信号流中发现异常信号

**入口**：侧边栏 → 事件生命周期 → **信号流** (`/signals`)

**操作过程**：
1. 打开信号流页面，查看原始信号列表
2. 使用筛选器过滤：选择来源 = `EDR`，级别 = `High`
3. 在列表中定位到目标信号（如：`SIG-20260509-0015`）
4. 查看信号详情：时间戳、来源设备、原始日志内容、协议类型
5. 点击 **「升级为告警」** 按钮，将信号升级为正式告警

**输入**：信号 ID、筛选条件
**输出**：升级后的告警编号（如 `ALT-20260509-0042`）

**注意事项**：
- 信号流中的每条记录代表一个原始安全事件，来自防火墙、EDR、IDS 等不同数据源
- 不是所有信号都需要升级为告警，AI 会自动筛选高价值信号
- 升级操作不可撤销，但可以在告警阶段标记为误报

#### 步骤 2：在 AI 分析工作台进行研判

**入口**：侧边栏 → 运营总览 → **AI 分析工作台** (`/ai-analysis`) 或直接点击告警卡片跳转

**操作过程**：
1. 进入 AI 分析工作台，看到升级后的告警已出现在告警列表中
2. 告警卡片显示关键信息：标题、严重级别、来源、AI 置信度、时间戳
3. 点击告警卡片展开 **AI 研判面板**，查看完整的 5 阶段推理链：
   - **事件发现**：检测到了什么异常（如 PowerShell 编码执行）
   - **证据关联**：多源证据交叉验证（VPN 日志 + EDR + 防火墙流量）
   - **模式匹配**：MITRE ATT&CK 技术映射（T1059.001 → T1071.001）
   - **置信度评估**：贝叶斯推理计算最终置信度（如 92%）
   - **处置决策**：推荐处置动作（冻结账号、隔离终端等）
4. 查看 **假设面板**：AI 生成的多个攻击假设及其置信度
5. 查看 **证据图谱**：实体关系图展示 IOC、资产、用户之间的关联
6. 确认研判结果后，点击 **「确认研判」** 或 **「提交处置」**

**输入**：告警 ID
**输出**：AI 研判结论（攻击类型、置信度、推荐处置动作）、假设列表、证据链

**注意事项**：
- AI 研判结论仅供参考，重要决策需人工复核
- 置信度 ≥ 85% 的告警会自动触发处置建议
- 可以在研判面板中添加人工备注和补充证据
- 信号流和告警列表支持 WebSocket 实时更新，新信号和告警自动推送至页面

#### 步骤 3：在处置响应页面执行处置

**入口**：侧边栏 → 事件生命周期 → **处置响应** (`/response`)

**操作过程**：
1. 进入处置响应页面，默认显示 **「自动处置」** 标签页
2. 在 **动作队列** 中找到待执行的处置动作：
   - 动作名称（如「冻结账号」「隔离主机」「封禁 IP」）
   - 目标对象（具体账号/IP/主机名）
   - 优先级（critical/high/medium/low）
   - 当前状态（pending/executing/completed/awaiting_approval）
   - AI 推理理由摘要
3. 展开动作卡片，查看 **完整推理链**（5 阶段详细推理过程）
4. 查看 **安全护栏信息**（审批要求、影响范围、回滚方案、审计记录）
5. 根据 **假设-动作映射** 面板了解该动作属于哪个攻击假设
6. 执行操作：
   - **pending 状态**：点击 **「执行」** 按钮立即执行
   - **awaiting_approval 状态**：点击 **「批准」** 或 **「拒绝」** 进行审批
   - **executing 状态**：可点击 **「取消」** 终止正在执行的动作
7. 执行后可在 **「执行记录」** 标签页查看历史处置日志
8. 如需回滚，在 **回滚记录** 区域执行回滚操作

**输入**：动作 ID、审批决定
**输出**：执行结果（成功/失败/超时）、执行日志、回滚选项

**注意事项**：
- critical 和 high 优先级的动作会在队列中置顶显示，带红色/橙色光晕效果
- 所有处置动作均记录完整审计日志，支持事后追溯
- 回滚操作需要填写原因并由授权人员执行

#### 流程图示

```
信号流(/signals) ──升级──→ AI分析工作台(/ai-analysis) ──研判──→ 处置响应(/response)
     │                        │                              │
     ▼                        ▼                              ▼
 原始信号列表             5阶段AI推理链                    动作队列执行
 筛选/搜索/升级          假设/证据/置信度                 审批/回滚/审计
```

### 3.2 工作流二：一次完整的攻击调查流程

**场景**：收到一条 VPN 异地登录的高危告警，需要进行完整的溯源调查。

#### 步骤 1：使用 AI 调查助手快速入门

**入口**：侧边栏独立入口或直接访问 `/ai-chat`

**操作过程**：
1. 打开 AI 调查助手页面，界面分为三个区域：
   - **左侧边栏**：历史对话列表 + 数据范围信息
   - **中间对话区**：聊天消息展示（支持滚动）
   - **底部输入框**：自然语言输入
2. 在快捷提问区域点击 **「分析最新告警」** 或直接输入：
   ```
   帮我分析一下最新的高危告警，给出攻击研判结论
   ```
3. AI 调查助手自动执行以下操作序列：
   - **工具调用**：调用 `query_auth_logs` 查询认证日志（显示工具调用卡片，含输入参数和返回结果）
   - **证据列表**：返回结构化证据，每条标注来源、时间戳、方向（支持/反对/中性）
   - **置信度更新**：展示调整前后的置信度变化条
   - **后续建议**：以建议芯片（Suggestion Chips）形式给出下一步行动选项
4. 基于返回的证据和建议，选择继续深入调查：
   - 点击建议芯片 **「联系员工确认是否本人操作」**
   - 或输入新的查询：`103.45.67.89 这个 IP 有什么威胁情报？`

**输入**：自然语言问题
**输出**：结构化回答 + 证据列表 + 工具调用记录 + 置信度更新 + 后续建议

**注意事项**：
- AI 调查助手支持 6 类快捷提问：告警分析、账号行为、IP 威胁查询、文件追溯、横向移动检测、态势总结
- 按 Enter 发送消息，Shift+Enter 换行
- 每条 AI 回复支持复制、点赞/点踩操作
- 对话历史保存在左侧边栏，支持新建对话和删除旧对话

#### 步骤 2：创建调查案件

**入口**：侧边栏 → 事件生命周期 → **调查/案件** (`/investigate`)

**操作过程**：
1. 在调查工作台中点击 **「新建案件」** 按钮
2. 填写案件基本信息：
   - 案件标题（如：「VPN 异地登录 - zhangsan 账号疑似失陷」）
   - 关联告警（从列表中选择或自动关联）
   - 优先级（P1-P4）
   - 负责人（分配给指定分析师）
   - 描述（初步调查发现）
3. 案件创建后进入 **案件详情页**，包含以下工作区：
   - **时间线视图**：按时间顺序排列的事件序列
   - **证据板**：拖拽式证据整理区域
   - **关联图谱**：实体关系可视化
   - **调查笔记**：自由文本记录区
4. 将 AI 调查助手提供的证据批量导入案件证据板

**输入**：案件信息表单
**输出**：案件编号（如 `CASE-2026-0042`）、案件工作台

#### 步骤 3：使用威胁狩猎深化分析

**入口**：侧边栏 → 增值模块 → **威胁狩猎** (`/hunting`)

**操作过程**：
1. 进入威胁狩猎页面，基于 MITRE ATT&CK 框架创建狩猎假设
2. 点击 **「新建假设」** 按钮，填写：
   - **假设名称**（如：「APT29 钓鱼攻击凭证窃取」）
   - **战术选择**（从 14 个 ATT&CK 战术中选择，如「初始访问」）
   - **技术选择**（对应战术下的具体技术，如「鱼叉式钓鱼 T1566.001」）
   - **初始置信度**和 **描述**
3. 假设创建后进入验证循环：
   - **IOC 收集**：添加相关 IOC（IP、域名、Hash、URL），系统自动查询威胁情报
   - **证据关联**：将信号流和调查案件中的证据关联到假设
   - **置信度更新**：根据新证据调整假设置信度
   - **状态流转**：验证中 → 已确认 / 已排除
4. 使用 ATT&CK 战术/技术矩阵视图，直观查看所有假设在框架中的分布

**输入**：狩猎假设定义、IOC 列表
**输出**：假设验证结果（已确认/已排除）、关联 IOC、威胁情报报告

**注意事项**：
- 威胁狩猎是主动式的威胁发现方法，区别于被动告警响应
- 每个假设必须映射到 ATT&CK 战术和技术 ID
- IOC 类型支持：IP、域名、Hash（MD5/SHA256）、URL
- 威胁情报查询整合了 VirusTotal、AlienVault OTX 等外部情报源

#### 步骤 4：完成调查并归档

1. 在调查案件中汇总所有发现
2. 生成调查报告（导出为 PDF）
3. 关联反馈学习条目（如果 AI 研判有偏差）
4. 将案件状态变更为 **「已关闭」**

### 3.3 工作流三：AI 驱动的自动化运营闭环

**场景**：配置 AI 自动处置策略，实现从告警到处置的全自动化运行，并通过策略自演化持续优化。

#### 步骤 1：配置自动处置策略

**入口**：处置响应 (`/response`) → **「处置策略」** 标签页

**操作过程**：
1. 查看预置的 4 条自动处置策略：

| 策略名称 | 触发条件 | 处置链 | 置信度阈值 |
|----------|----------|--------|------------|
| 账号失陷自动处置 | 置信度 ≥ 85% | 冻结账号 → 重置密码 → 隔离终端 | 85% |
| C2 通信自动阻断 | 置信度 ≥ 90% | 封禁 IP → 隔离终端 → 通知团队 | 90% |
| 暴力破解自动防御 | 失败次数 ≥ 50 | 封禁 IP → 锁定账号 → 启用 MFA | 95% |
| 数据外泄自动遏制 | 外传 ≥ 100MB | 阻断连接 → 取证 → 通知 DLP | 80% |

2. 点击策略卡片查看详细配置：
   - **触发条件**：满足何种条件时激活
   - **处置动作链**：按序执行的多个动作
   - **置信度阈值**：AI 置信度达到多少时自动执行
   - **人工审批节点**：哪些环节需要人工介入
3. 点击 **「编辑策略」** 自定义或创建新策略
4. 在 **策略演化** 页面可查看 AI 基于历史处置效果自动优化策略参数

**输入**：策略配置（触发条件、动作链、阈值、审批节点）
**输出**：生效的自动处置策略

#### 步骤 2：配置 SOAR 剧本

**入口**：侧边栏 → 增值模块 → **自动化编排** (`/workflows`)

**操作过程**：
1. 进入 SOAR 剧本引擎页面，查看已有 Playbook 列表
2. 每个 Playbook 显示：
   - 名称和描述
   - 触发条件
   - 步骤数量
   - 总执行次数
   - 最近执行时间
   - 启用/禁用状态
3. 点击 Playbook 查看 **可视化流程图**：
   - **触发节点**（Trigger）：什么条件启动剧本
   - **条件节点**（Condition）：分支判断逻辑
   - **动作节点**（Action）：具体执行的操作
   - **审批节点**（Approval）：需人工确认的环节
   - 节点间连线表示执行流向
4. 点击 **「新建剧本」** 创建自定义 Playbook：
   - 选择触发器类型（定时器/告警/Webhook/手动）
   - 拖拽添加节点构建流程
   - 配置每个节点的参数
   - 设置错误处理和重试策略
5. 在 **执行记录** 标签页查看历史运行情况

**输入**：Playbook 定义（节点、连线、参数）
**输出**：可执行的自动化剧本

#### 步骤 3：建立策略自演化闭环

**入口**：侧边栏 → AI 能力 → **策略自演化** (`/learning`)

**操作过程**：
1. 进入策略自演化系统，查看 4 个标签页的数据：

**标签页 1 — 知识反馈概览**：
- 今日反馈数、确认率、平均置信度提升等统计卡片
- 近期反馈列表（案件 ID、攻击类型、人工操作、置信度变化）

**标签页 2 — 反馈学习明细**：
- 每条反馈的详细信息：原始 AI 评估 vs 人工修正
- 学习结果：AI 如何根据反馈调整判断
- 类型区分：确认类（confirmed）vs 争议类（disputed）

**标签页 3 — 推理校正**：
- 权重调整记录：哪些推理规则的权重被修改
- 模式改进：识别到的新攻击模式
- 校正前后置信度对比

**标签页 4 — AI 成长曲线**：
- 准确率随时间的变化趋势图
- 关键里程碑事件（首次超过 95%、处理第 1000 条等）
- 各攻击类型的识别准确率分布

2. 对于每条 AI 研判结果，操作人员可以：
   - **确认**：同意 AI 的判断（提高此类模式的权重）
   - **争议**：不同意 AI 的判断（降低权重并提供正确答案）
   - **补充**：添加额外上下文信息

3. 反馈提交后，AI 模型通过 Q-Learning 算法在下一次迭代中自动吸收这些反馈，优化处置策略参数

**输入**：人工反馈（确认/争议 + 备注）
**输出**：AI 模型参数微调、准确率提升、策略参数优化

**闭环示意**：

```
告警产生 → AI研判 → 自动/人工处置 → 人工反馈 → 模型优化 → 更精准的研判 ↑
    ↑                                                                    │
    └────────────────── 循环往复，持续进化 ──────────────────────────────┘
```

### 3.4 工作流四：多租户管理与团队协作

**场景**：MSP/MSSP 服务商需要管理多个客户租户，每个租户拥有独立的数据隔离、用户体系和运营视图。

#### 步骤 1：创建与管理租户

**入口**：侧边栏 → 租户与订阅 → **多租户管理** (`/tenants`)

**操作过程**：
1. 进入多租户管理页面，查看租户列表
2. 点击 **「创建租户」** 按钮，填写租户信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| 租户名称 | 客户公司名称 | 某金融科技公司 |
| 租户标识 | 唯一标识符（字母数字） | fintech-001 |
| 管理员邮箱 | 租户超级管理员邮箱 | admin@fintech.com |
| 数据保留策略 | 日志保留天数 | 90 天 |
| 资源配额 | 告警处理上限、用户数上限 | 10000/月, 20 用户 |
| 订阅计划 | 基础版/专业版/企业版 | 专业版 |

3. 创建后系统自动完成以下操作：
   - 创建独立的数据库 Schema（RLS 行级安全隔离）
   - 发送邀请邮件给租户管理员
   - 初始化租户的默认角色和权限模板
   - 分配存储和计算资源配额

**输入**：租户注册信息
**输出**：新租户环境（含独立数据库隔离、管理员账号）

#### 步骤 2：配置租户隔离策略

**入口**：多租户管理 → 选择租户 → **隔离策略** 标签页

**操作过程**：
1. 查看当前租户的数据隔离级别：

| 隔离级别 | 说明 | 适用场景 |
|----------|------|----------|
| **Schema 隔离** | 每个租户独立数据库 Schema | 金融、医疗等强合规行业 |
| **RLS 行级隔离** | 同一表内通过 tenant_id 字段隔离 | 中小企业、标准合规需求 |
| **混合模式** | 核心数据 Schema 隔离 + 共享数据 RLS | 大型 MSP 服务商 |

2. 配置租户的访问控制策略：
   - **IP 白名单**：限制租户管理员登录 IP 范围
   - **SSO 强制登录**：要求使用客户自有 IdP 认证
   - **API 访问控制**：限制租户 API 调用权限
3. 配置资源配额和告警阈值

**输入**：隔离策略配置
**输出**：生效的租户隔离规则

#### 步骤 3：租户内用户与权限管理

**入口**：多租户管理 → 选择租户 → **用户管理** 标签页

**操作过程**：
1. 租户管理员登录后，在自身租户范围内管理用户
2. 支持的角色体系：

| 角色 | 权限范围 | 适用人员 |
|------|----------|----------|
| 租户超级管理员 | 租户内全部功能权限 | 客户安全负责人 |
| 租户管理员 | 除系统设置外的全部权限 | 客户高级安全工程师 |
| 租户分析师 | 运营总览 + 事件 lifecycle + AI 能力 | 客户安全分析师 |
| 租户只读用户 | 仅查看权限 | 客户审计员 |

3. 支持批量导入用户（CSV 格式）
4. 支持跨租户的 **团队协作** 功能：
   - **案件共享**：将特定案件共享给其他租户的分析师
   - **联合调查**：多个租户的分析师协同调查跨租户攻击
   - **知识库共享**：选择性地共享威胁情报和处置经验

**输入**：用户信息、角色分配
**输出**：租户内用户体系

#### 步骤 4：查看租户运营概览

**入口**：多租户管理 → **运营概览** 标签页

**操作过程**：
1. 查看所有租户的聚合运营数据：

| 指标 | 说明 |
|------|------|
| 活跃租户数 | 当前在线的租户数量 |
| 今日告警总量 | 所有租户的告警汇总 |
| 平均 MTTR | 各租户的平均响应时间 |
| 租户健康度 | 各租户的系统健康评分 |
| 配额使用率 | 各租户的资源使用情况 |

2. 支持按租户筛选查看详细运营数据
3. 支持导出多租户运营报告

**输入**：时间范围、租户筛选条件
**输出**：多租户运营数据看板

### 4.1 AI 分析工作台（/ai-analysis）

**功能定位**：平台的核心 AI 研判界面，所有安全告警在此汇聚并进行智能分析。

**入口位置**：侧边栏 → 运营总览 → AI 分析工作台

#### 4.1.1 页面布局

页面采用 **三栏式布局**：

```
┌─────────────┬──────────────────┬─────────────────┐
│             │                  │                 │
│  告警列表    │   AI 研判面板     │   假设面板      │
│  (左侧)     │   (中间主区域)    │   (右侧)        │
│             │                  │                 │
│ • 告警卡片   │ • 5阶段推理链     │ • 攻击假设列表   │
│ • 筛选器    │ • 证据详情        │ • 置信度指示     │
│ • 搜索      │ • MITRE 映射      │ • 关联动作       │
│             │ • 置信度评分      │                 │
└─────────────┴──────────────────┴─────────────────┘
```

#### 4.1.2 告警列表（左侧栏）

**功能**：展示所有待分析和已分析的告警卡片。

**字段说明**：

| 字段 | 描述 | 示例 |
|------|------|------|
| 告警 ID | 唯一标识符 | ALT-20260509-0042 |
| 标题 | 告警简要描述 | 主机 WIN-02 检测到 PowerShell 编码执行 |
| 严重级别 | Critical / High / Medium / Low / Info | 用颜色标签区分 |
| 来源 | 数据源类型 | EDR / 防火墙 / VPN网关 |
| AI 置信度 | AI 判定为真实攻击的概率 | 92%（进度条可视化） |
| 时间戳 | 告警产生时间 | 2026-05-09 08:18:05 |
| 状态 | analyzing / confirmed / false_positive / escalating | 分析中/已确认/误报/升级中 |

**交互操作**：
- **单击**告警卡片 → 在中间区域加载对应的 AI 研报面板
- **筛选器**：按严重级别、来源、时间范围、状态筛选
- **搜索框**：模糊搜索告警标题和 ID
- **排序**：按时间、置信度、严重级别排序

#### 4.1.3 AI 研判面板（中间主区域）

**功能**：展示 AI 对选中告警的完整推理分析过程。

**五阶段推理链详解**：

| 阶段 | 图标 | 名称 | 说明 |
|------|------|------|------|
| Phase 1 | 🔍 Search | 事件发现 | 描述检测到的异常事件是什么 |
| Phase 2 | 🔗 Link2 | 证据关联 | 将来自多个数据源的证据进行交叉关联 |
| Phase 3 | 🎯 Target | 模式匹配 | 与已知攻击模式和 MITRE ATT&CK 技术进行匹配 |
| Phase 4 | 📊 BarChart3 | 置信度评估 | 基于贝叶斯推理计算综合置信度 |
| Phase 5 | 💡 Lightbulb | 处置决策 | 基于置信度和策略推荐处置动作 |

**每个推理阶段的展示内容包括**：
- 阶段标题和编号
- 自然语言描述
- 证据列表（每条证据含：来源、时间戳、详情、方向）
- 该阶段的结论
- （可选）MITRE ATT&CK 技术引用编号
- （可选）置信度增量

**v3.0 新增 — 推理链可视化**：
- 推理链以 **流程图形式** 可视化展示，每个阶段为一个节点，节点间箭头表示推理流向
- 每个节点可展开查看该阶段的详细推理过程和中间结论
- 支持 **推理链导出** 为图片或 PDF，便于报告和审计
- 推理链中的每个证据点均可点击跳转到原始数据源详情
- 支持 **对比模式**：同时展示 AI 推理链与人工修正后的推理链差异

**证据方向说明**：

| 方向 | 含义 | 颜色 |
|------|------|------|
| supports | 支持攻击假设 | 绿色 |
| contradicts | 反对攻击假设 | 红色 |
| neutral | 中性，既不支持也不反对 | 灰色 |

#### 4.1.4 假设面板（右侧栏）

**功能**：展示 AI 为当前告警生成的多个攻击假设。

**字段说明**：

| 字段 | 描述 |
|------|------|
| 假设 ID | H-A、H-B、H-C 等标识 |
| 假设标签 | 攻击类型描述（如「账号被盗」、「内部人员威胁」、「误报」） |
| 置信度 | 该假设成立的概率（百分比） |
| 关联动作 | 属于此假设的处置动作列表及状态 |
| 进度条 | 显示已完成/总数/执行中/待处理/等待审批的数量 |

**视觉指示**：
- 置信度 ≥ 80%：绿色高亮，带发光效果
- 置信度 50-80%：黄色/琥珀色
- 置信度 < 50%：灰色

#### 4.1.5 证据图谱

**功能**：以图形化方式展示证据之间的实体关系。

**节点类型**：
- **IOC 节点**（IP、域名、Hash）：菱形，红色边框
- **资产节点**（主机名、服务器）：方形，蓝色边框
- **用户节点**（账号）：圆形，绿色边框
- **事件节点**（告警、信号）：六边形，橙色边框

**连线含义**：表示两个实体之间存在关联关系（如「登录自」「访问过」「通信」等）。

#### 4.1.6 注意事项

- AI 研判结论的置信度是一个概率值，不是确定性判断
- 当存在矛盾证据时（同时有支持和反对的证据），AI 会降低置信度并建议人工介入
- 所有推理过程完全透明可追溯，每个结论都有对应的证据支撑
- 研判面板支持导出为 PDF 报告

#### 4.20.5 AI 调用的内部工具

当用户提问时，AI 调查助手会自动调用以下内部工具：

| 工具名称 | 功能 | 输入 | 输出示例 |
|----------|------|------|----------|
| query_auth_logs | 查询认证日志 | target, timeframe | 找到 N 条异常登录记录 |
| threat_intel_lookup | 威胁情报查询 | ioc, sources | 从 N 个情报源获取结果 |
| query_file_access | 文件访问查询 | target, timeframe, scope | 检索到 N 条访问记录 |

| aggregate_security_posture | 安全态势聚合 | date, scope | 生成态势摘要 |

每个工具调用都以 **Tool Call 卡片** 形式展示，包含：
- 工具名称和图标
- 执行耗时
- 输入参数（JSON 格式，可展开）
- 返回结果文本

#### 4.20.6 v3.0 增强 — 多轮工具调用与报告生成

**多轮工具调用（Multi-turn Tool Calling）**：
- AI 支持在单次对话中 **自动串联多个工具调用**，无需用户逐一手动触发
- 例如：分析告警时，AI 可依次调用 `query_auth_logs` → `threat_intel_lookup` → `lateral_movement_scan`，形成完整的调查链路
- 工具调用链以 **流程图形式** 可视化展示，每个节点显示工具名称、输入输出摘要
- 支持 **中断与修正**：用户可在工具调用链执行过程中插入指令，调整后续调用方向

**报告生成**：
- AI 调查助手支持将对话内容一键生成为 **结构化调查报告**
- 报告格式：PDF / Markdown / HTML
- 报告内容自动组织：调查背景 → 工具调用记录 → 证据列表 → 分析结论 → 建议措施
- 生成的报告可直接关联到调查案件或导出存档

- **折叠侧边栏**：点击标题栏的折叠按钮扩大对话区域

- **附带附件**：点击「附带附件」上传文件供 AI 分析

#### 4.20.8 使用示例

**示例 1：分析 VPN 异常登录**

```

   置信度: 92% → 45%（因出差申请反对证据下调）
```

### 4.22 策略演化（/strategy-evolution）

**功能定位**：处置策略的自动演化与优化引擎，基于 Q-Learning 算法持续优化处置策略参数。

**入口位置**：侧边栏 → 增值模块 → 策略演化

#### 4.22.1 页面布局

| 区域 | 内容 |
|------|------|
| **演化概览** | 当前策略版本、演化次数、平均置信度提升、策略效果评分 |
| **策略列表** | 所有可演化的策略及其当前参数配置 |
| **演化历史** | 每次演化的详细记录（版本号、参数变更、效果对比） |
| **效果对比** | 演化前后的关键指标对比（准确率、处置率、误报率） |

#### 4.22.2 核心功能

**策略演化引擎**：
- 基于 **Q-Learning 强化学习** 算法，根据历史处置效果自动调整策略参数
- 每次演化生成 **策略版本**，支持版本对比和回滚
- 演化参数包括：置信度阈值、处置动作优先级、审批规则、回滚策略

**演化指标**：

| 指标 | 说明 |
|------|------|
| 策略版本 | 当前策略的版本号（v1, v2, ...） |
| 演化次数 | 累计自动演化次数 |
| 平均置信度提升 | 演化后 AI 研判置信度的平均提升百分比 |
| 策略效果评分 | 综合评分（0-100），基于准确率、处置率、误报率加权计算 |
| 回滚率 | 演化后被回滚的比例，越低表示演化效果越好 |

**操作功能**：
- **查看演化建议**：AI 自动生成策略优化建议，含预期效果评估
- **应用演化**：手动确认应用 AI 推荐的策略参数调整
- **版本对比**：并排对比两个版本的策略参数和效果指标
- **回滚**：将策略回滚到指定历史版本
- **自动演化开关**：开启后 AI 自动执行策略优化（需配置审批规则）

#### 4.22.3 演化流程

```
当前策略 → AI 分析历史处置效果 → 生成优化建议 → 人工审核 → 应用新版本 → 效果追踪
    ↑                                                                       │
    └────────────────────── 持续演化循环 ────────────────────────────────────┘
```

- **查看订单历史**：所有订阅变更和付款记录

## 第五章 AI 功能使用指南

### 5.1 AI 研判引擎原理

#### 5.1.1 五阶段推理模型

SecMind 的 AI 研判引擎采用 **五阶段链式推理模型**，模拟资深安全分析师的思维过程：

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  事件发现  │ → │ 证据关联  │ → │ 模式匹配  │ → │ 置信度评估 │ → │ 处置决策  │
│ Phase 1  │   │ Phase 2  │   │ Phase 3  │   │ Phase 4  │   │ Phase 5  │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
```

**Phase 1 — 事件发现（Event Discovery）**
- 从原始信号中提取关键事件要素
- 识别异常行为的时空特征
- 输出：「检测到了 X」

**Phase 2 — 证据关联（Evidence Correlation）**
- 跨数据源关联相关证据
- 时序对齐，建立因果关系
- 评估每条证据的方向（支持/反对/中性）
- 输出：「N 条证据支持，M 条反对」

**Phase 3 — 模式匹配（Pattern Matching）**
- 与已知攻击模式库比对
- 映射到 MITRE ATT&CK 技术 ID
- 关联外部威胁情报
- 输出：「匹配 TXXXX → TXXXX 攻击链」

**Phase 4 — 置信度评估（Confidence Assessment）**
- 贝叶斯推理加权计算
- 考虑证据时效性和来源可靠性
- 处理矛盾证据的降权
- 输出：「综合置信度 XX%」

**Phase 5 — 处置决策（Disposal Decision）**
- 基于置信度匹配处置策略
- 评估业务影响
- 确定是否需要人工审批
- 输出：「推荐执行 X 动作」

#### 5.1.2 证据方向体系

AI 在推理过程中对每条证据标注方向：

| 方向 | 含义 | 权重影响 |
|------|------|----------|
| **Supports（支持）** | 证据表明攻击假设可能成立 | + 增加置信度 |
| **Contradicts（反对）** | 证据表明攻击假设可能不成立 | - 降低置信度 |
| **Neutral（中性）** | 证据与假设无明显相关性 | 无直接影响 |

**特殊处理**：当存在强反对证据时，即使支持证据众多，AI 也会显著降低置信度并建议人工介入。例如：VPN 异地登录 + C2 通信等多条支持证据，但如果 HR 出差申请构成强反对证据，置信度可能从 92% 下调至 45%。

#### 5.1.3 多假设竞争机制

对于每个告警，AI 同时生成 **多个竞争假设**，而非单一结论：

```
告警: VPN 异地登录 + PowerShell 异常执行
    │
    ├── H-A: 账号被盗 (置信度 92%) ← 最可能的假设
    │     ├── 冻结账号 (pending)
    │     ├── 隔离主机 (executing)
    │     ├── 封禁 IP (completed)
    │     └── ...
    │
    ├── H-B: 内部人员威胁 (置信度 54%) ← 不能排除
    │     ├── 监控用户 (pending)
    │     └── 审查日志 (pending)
    │
    └── H-C: 误报 (置信度 12%) ← 可能性低
```

这种多假设机制避免了 AI 的"过度自信"，确保不会遗漏低概率但高风险的可能性。

### 5.2 AI 调查助手对话式操作

#### 5.2.1 设计理念

AI 调查助手采用 **对话式交互 + 工具调用** 的混合架构：

- **自然语言理解**：用户可以用日常语言描述调查需求
- **工具调用（Function Calling）**：AI 自动选择合适的内部工具查询数据
- **结构化输出**：返回证据列表、置信度评估、后续建议
- **连续对话**：支持多轮追问和逐步深入分析

#### 5.2.2 对话最佳实践

**✅ 推荐用法**：

| 做法 | 示例 | 效果 |
|------|------|------|
| 明确指定目标 | "分析 zhangsan 账号的异常登录" | AI 直接定位到目标 |
| 指定时间范围 | "查询过去 7 天的文件访问" | 缩小查询范围，结果更精确 |
| 组合多个问题 | "这个 IP 的威胁情报 + 关联的内网通信" | 一次性获得全面视图 |
| 基于建议继续 | 点击 AI 返回的建议芯片 | 沿着 AI 推荐的最优路径深入 |

**❌ 避免用法**：

| 做法 | 问题 |
|------|------|
| 过于宽泛的问题 | "帮我看看有什么安全问题" → 返回过多信息 |
| 缺少上下文 | "这个有问题吗？" → AI 不知道指什么 |
| 期望实时指令 | "现在帮我封禁这个 IP" → AI 调查助手仅做分析，不执行处置 |

#### 5.2.3 对话 vs 传统 UI 的选择

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| 快速了解某个 IOC | AI 调查助手 | 一句话搞定 |
| 深度调查一个案件 | 调查/案件页面 | 结构化的证据组织更好 |
| 批量处置操作 | 处置响应页面 | 可视化队列和审批流程 |
| 日常态势巡检 | 仪表盘 | KPI 和趋势一目了然 |
| 探索性分析 | AI 调查助手 | 自然语言灵活探索 |

### 5.3 策略自演化系统

#### 5.3.1 为什么需要策略自演化

AI 研判引擎虽然经过训练，但在面对特定企业的独特环境时可能出现偏差：

- **误报**：正常业务行为被误判为攻击（如合法的远程办公）
- **漏报**：新型攻击手法未被识别
- **置信度偏差**：判断方向正确但置信度过高或过低

策略自演化机制让每次人工纠正都成为 AI 进化的养料，通过 Q-Learning 强化学习算法持续优化处置策略参数。

#### 5.3.2 反馈类型与影响

| 反馈类型 | 操作 | 对模型的影响 |
|----------|------|--------------|
| **确认（Confirmed）** | 点击「确认攻击」 | 提高此类攻击模式的特征权重 |
| **争议（Disputed）** | 点击「标记误报」+ 提供正确答案 | 降低误判模式的权重，学习正确模式 |
| **补充（Supplement）** | 添加额外的上下文信息 | 丰富决策依据，减少未来类似误判 |

#### 5.3.3 学习效果度量

在策略自演化的 **AI 成长曲线** 标签页中，可以观察：

- **总体准确率趋势**：应呈上升趋势
- **分攻击类型准确率**：某些类型可能有系统性偏差
- **里程碑事件**：如「首次突破 95%」「累计处理 1000 条反馈」
- **置信度校准曲线**：预测置信度与实际准确率的吻合程度
- **策略演化效果**：每次演化前后的处置效果对比

### 5.4 AI 自动处置策略

#### 5.4.1 置信度-优先级映射

AI 根据研判置信度自动确定处置动作的优先级：

| 置信度范围 | 优先级 | 自动化程度 | 说明 |
|------------|--------|------------|------|
| ≥ 80% | Critical | 高度自动 | 自动执行，异步通知 |
| 50% - 79% | High | 半自动 | 推荐执行，需快速审批 |
| 20% - 49% | Medium | 人工主导 | 仅建议，人工决定 |
| < 20% | Low | 仅记录 | 不触发处置，仅记录备查 |

#### 5.4.2 安全护栏机制

每个自动处置动作都配备 **四重安全护栏**：

| 护栏 | 内容 | 示例 |
|------|------|------|
| **审批护栏** | 是否需要人工审批及条件 | "自动阈值已满足，人工可在审计台一键撤销" |
| **影响护栏** | 预估的业务影响范围 | "影响 1 个账号，预计中断 VPN、SSO 与邮箱会话" |
| **回滚护栏** | 如何撤销已执行的处置 | "支持解冻账号、恢复 MFA 与重新下发临时凭证" |
| **审计护栏** | 完整的操作记录 | "记录 AI 推理链、触发策略、执行人与执行结果" |

#### 5.4.3 回滚机制

所有自动处置动作都支持回滚：

1. 在 **处置响应 → 执行记录** 标签页找到目标操作
2. 点击 **「回滚」** 按钮
3. 填写回滚原因（必填）
4. 系统执行反向操作并记录回滚日志
5. 回滚操作本身也会被审计记录

> **注意**：回滚操作需要授权角色才能执行，且回滚原因会被记录用于后续 AI 学习。

### 5.5 AI 可解释性与推理链

#### 5.5.1 推理链可视化

v3.0 引入了 **推理链可视化** 功能，将 AI 的研判过程以流程图形式直观展示：

- **节点化展示**：5 阶段推理链的每个阶段为一个节点，节点间箭头表示推理流向
- **可展开详情**：每个节点可展开查看该阶段的详细推理过程、中间结论和置信度变化
- **证据溯源**：推理链中的每个证据点均可点击跳转到原始数据源详情
- **对比模式**：同时展示 AI 推理链与人工修正后的推理链差异，便于审计和培训

#### 5.5.2 推理链导出

- 支持将完整推理链导出为 **图片（PNG/SVG）** 或 **PDF 报告**
- 导出的推理链包含：告警基本信息、5 阶段推理过程、证据列表、置信度变化曲线
- 适用于安全审计、事件复盘和团队培训场景

#### 5.5.3 可解释性指标

| 指标 | 说明 |
|------|------|
| **推理完整性** | 推理链是否覆盖了所有必要的分析维度 |
| **证据充分性** | 每个结论是否有足够的证据支撑 |
| **矛盾处理** | AI 是否正确处理了矛盾证据 |
| **置信度校准** | AI 的置信度评估是否与实际准确率一致 |

#### 5.5.4 人工修正与审计

- 分析师可在推理链的任意阶段进行 **人工修正**（添加/删除证据、调整置信度）
- 所有修正操作被完整记录在审计日志中
- 修正后的推理链可作为 **训练样本** 输入策略自演化系统

### 5.6 多模型智能路由

#### 5.6.1 设计理念

SecMind v3.0 支持 **多模型智能路由（Model Router）** 机制，根据任务类型、复杂度、实时性要求等因素，自动选择最优的 LLM 模型：

- **成本优化**：简单任务使用轻量模型，复杂任务使用高性能模型
- **性能平衡**：实时性要求高的任务路由到低延迟模型
- **故障转移**：主模型不可用时自动切换到备用模型
- **本地优先**：支持本地部署的开源模型，满足数据安全合规要求

#### 5.6.2 支持的模型

| 模型 | 类型 | 适用场景 | 特点 |
|------|------|----------|------|
| **GPT-4o** | 云端大模型 | 复杂研判、深度分析 | 推理能力强，延迟较高 |
| **Claude Sonnet** | 云端大模型 | 证据分析、报告生成 | 长上下文，结构化输出 |
| **本地 LLM** | 本地开源模型 | 实时研判、基础分析 | 低延迟，数据不出域 |
| **专用小模型** | 本地微调模型 | 分类、标签、模式匹配 | 高吞吐，低成本 |

#### 5.6.3 路由策略

| 路由策略 | 说明 | 配置方式 |
|----------|------|----------|
| **基于任务类型** | 研判、分析、报告等不同任务路由到不同模型 | 系统预设 |
| **基于置信度阈值** | 低置信度任务自动升级到更强模型 | 可自定义 |
| **基于延迟要求** | 实时任务路由到低延迟模型 | 系统预设 |
| **基于成本预算** | 在预算范围内选择最优模型组合 | 可自定义 |
| **手动指定** | 用户手动选择使用的模型 | 按需选择 |

#### 5.6.4 模型配置

- 在 **系统设置 → AI 配置** 中管理模型路由规则
- 支持为不同租户配置不同的模型策略
- 提供模型调用统计和成本分析面板
- 支持模型 A/B 测试，对比不同模型的效果

### 6.1 数据源接入配置

#### 6.1.1 Syslog 接入（最常用）

适用于防火墙、WAF、IDS/IPS、NAC 等支持 Syslog 协议的设备。

**设备端配置**（以 FortiGate 为例）：

```
config log syslogd setting
    status enable
    server "<SecMind-Server-IP>"
    port 514
    format cef
    facility local7
    source-ip "<FortiGate-Mgmt-IP>"
end
```

**平台端配置**：
1. 进入 **数据源** 页面
2. 点击「+ 添加数据源」
3. 类型选择对应设备类型（如「防火墙」）
4. 协议选择 `Syslog`
5. 端口填写 `514`（或自定义端口）
6. 日志格式选择 `CEF`
7. 传输方向选择 `push`
8. 保存后平台开始在指定端口监听

**验证方法**：保存后在数据源列表中查看状态是否变为 🟢 在线，并检查「最后同步」时间是否更新。

#### 6.1.2 API 接入

适用于 EDR、SIEM、云平台等支持 API 拉取的数据源。

**配置步骤**：
1. 传输方向选择 `pull`
2. 填写 API 地址（如 `https://edr.example.com/api/v1/logs`）
3. 配置认证凭证（API Key / OAuth / 证书）
4. 设置拉取间隔（建议 60s - 300s）
5. 保存并测试连接

#### 6.1.3 Agent 接入

适用于需要深度端点可见性的 EDR 解决方案。

**部署步骤**：
1. 在 **数据源** 页面下载 Agent 安装包
2. 在目标终端上安装 Agent
3. Agent 自动注册并开始上报数据
4. 在平台中查看 Agent 在线状态和健康度

### 6.2 Webhook 配置

Webhook 用于将 SecMind 的事件实时推送到第三方系统。

#### 6.2.1 配置入口

**集成中心** → 选择目标集成 → 编辑 → Webhook 配置

#### 6.2.2 Webhook Payload 格式

当配置的事件触发时，SecMind 会向目标 URL 发送 POST 请求：

```json
{
  "event_type": "alert.confirmed",
  "event_id": "evt_abc123",
  "timestamp": "2026-05-09T08:30:00Z",
  "data": {
    "alert_id": "ALT-20260509-0042",
    "title": "主机 WIN-02 检测到 PowerShell 编码执行",
    "severity": "critical",
    "confidence": 92,
    "hypothesis": "账号被盗",
    "assigned_to": "张明远"
  }
}
```

#### 6.2.3 签名验证

为确保 Webhook 请求的真实性，每个请求头中都包含 `X-SecMind-Signature` 签名：

```
X-SecMind-Signature: sha256=hex(hmac_sha256(webhook_secret, body))
```

接收方应验证签名以确保请求未被篡改。

### 6.3 SOAR 剧本编排

#### 6.3.1 创建自定义 Playbook

   - 从左侧节点面板拖拽节点到画布

6. 测试通过后点击 **「发布」**

#### 6.3.2 节点参数配置示例

**触发节点（Trigger）**：
```yaml
type: alert_created
filters:
```

```yaml
expression: "{{ confidence }} >= 85 && {{ impact_level }} != 'critical_business'"
branches:
```

```yaml
type: freeze_account
```

```yaml
approvers: ["security_team_lead"]

escalation: security_manager
on_timeout: auto_approve  # or auto_reject
```

| 实践 | 说明 |
|------|------|
| 幂等性设计 | 确保重复执行不会产生副作用 |
| 超时保护 | 每个节点设置合理的超时时间 |
| 错误处理 | 为每个分支定义失败时的回退动作 |

## 第八章 常见问题 FAQ

### Q1：AI 研判的置信度是什么意思？应该信任多高的置信度？

**A**：置信度是 AI 对"这是一个真实攻击"的概率估计，取值 0-100%。一般建议：
- **≥ 85%**：高度可信，可以基本依赖 AI 的判断
- **65%-84%**：中等可信，建议快速人工复核
- **< 65%**：低可信度，必须人工深入调查

置信度不是绝对准确的，它会随着反馈学习不断校准。初期建议对所有 ≥ 80% 的研判都进行人工复核。

### Q2：如何降低误报率？

**A**：降低误报率的关键在于**持续的反馈学习**：
1. 定期（建议每天）查看 **反馈学习** 页面的待反馈列表
2. 对每条误报点击「标记误报」并提供正确的分类
3. 特别关注反复出现的误报模式，考虑在 **知识库** 中添加例外规则
4. 在 **系统设置 → AI 配置** 中适当提高自动处置的置信度阈值
5. 经过 2-4 周的持续反馈，误报率通常会显著下降

### Q3：自动处置会不会造成业务中断？

**A**：平台设计了多重安全机制来防止这种情况：
1. **置信度门槛**：只有高置信度（默认 ≥ 85%）才触发自动处置
2. **影响评估**：每个处置动作执行前会评估业务影响
3. **人工审批**：中高风险动作需要人工批准
4. **即时回滚**：所有处置动作都可以在一键回滚
5. **渐进式上线**：建议先开启「通知模式」（只通知不执行），观察一周后再开启自动执行

### Q4：数据源显示离线怎么办？

**A**：按以下步骤排查：
1. **检查网络连通性**：从 SecMind 服务器 ping 数据源 IP
2. **检查端口**：`telnet <数据源 IP> <端口>` 确认端口可达
3. **检查防火墙规则**：确保 SecMind 服务器的入站规则允许数据源端口的流量
4. **检查设备端配置**：确认设备端的日志转发目标地址和端口正确
5. **检查协议匹配**：平台配置的协议（Syslog/CEF/API）必须与设备端一致
6. **查看平台日志**：`backend/logs/` 目录下可能有详细的连接错误信息

### Q5：AI 调查助手和 AI 分析工作台有什么区别？该用哪个？

**A**：两者的定位不同：

| 维度 | AI 调查助手 | AI 分析工作台 |
|------|-------------|--------------|
| **交互方式** | 对话式（自然语言） | 结构化界面（点击操作） |
| **适用场景** | 探索性分析、快速查询 | 深度研判、正式调查 |
| **输出形式** | 对话消息 + 证据列表 | 完整推理链 + 假设面板 |
| **处置能力** | 仅分析和建议 | 可直接触发处置 |
| **协作性** | 单人对话为主 | 适合团队协作办案 |

**简单原则**：想快速了解某个问题 → 用 AI 调查助手；要做正式的调查和处置 → 用 AI 分析工作台。

### Q6：如何添加新的 ATT&CK 战术/技术覆盖？

**A**：SecMind 的 ATT&CK 覆盖基于内置的战术技术库。如需扩展：
1. 在 **威胁狩猎** 页面创建假设时，可以选择任意已有的战术/技术
2. 如果需要的战术/技术不在列表中，可以通过 **系统设置 → 知识库管理** 添加自定义战术/技术定义
3. 添加后即可在狩猎假设中选择使用
4. 也支持通过 **集成中心 → TI 平面** 同步最新的 ATT&CK 数据

### Q7：反馈学习的效果多久能体现？

**A**：这取决于反馈的量和质量：
- **少量反馈（< 50 条）**：1-2 周内可以看到局部改善
- **中等反馈（50-200 条）**：2-4 周可以看到明显的准确率提升
- **大量反馈（200+ 条）**：1-2 个月可以达到稳定的高准确率（> 95%）

关键是反馈的**质量**比数量更重要——每条反馈都应该经过仔细核实。可以在 **反馈学习 → AI 成长曲线** 中实时跟踪学习效果。

### Q8：支持多租户/多部门隔离吗？

**A**：当前版本基于用户角色进行权限控制。多租户隔离可通过以下方式实现：
1. 利用 **用户管理** 中的角色体系，为不同部门创建不同的角色
2. 利用 **资产管理** 中的「所属业务线」字段进行逻辑隔离
3. 在 **数据源** 配置时按部门分配不同的数据源
4. 后续版本将原生支持多租户数据隔离

### Q9：如何导出调查报告？

**A**：有两种方式：
1. **从调查/案件页面**：打开案件详情 → 点击「导出报告」→ 选择格式（PDF/Word/Excel）
2. **从报告中心**：选择「事件报告」类型 → 选择案件 → 生成报告

导出的报告包含：案件信息、时间线、证据清单、AI 研判结论、处置记录、调查结论。

### Q10：平台支持哪些语言的 API？

**A**：SecMind 后端基于 FastAPI，提供 RESTful API，支持任何编程语言的 HTTP 调用。主要接口包括：
- `GET /api/alerts` — 获取告警列表
- `POST /api/alerts/{id}/analyze` — 触发 AI 研判
- `GET /api/cases` — 获取案件列表
- `POST /api/disposal/execute` — 执行处置动作
- `POST /api/ai-chat` — AI 调查助手接口

完整的 API 文档可通过访问 `http://your-backend:8000/docs` 查看（Swagger UI）。

### Q11：v3.0 的推理链可视化是什么？有什么作用？

**A**：推理链可视化是 v3.0 引入的核心功能，将 AI 的研判过程以流程图形式直观展示：
1. **透明可追溯**：每个研判结论都有完整的推理路径，分析师可以逐阶段审查 AI 的思考过程
2. **便于审计**：推理链可导出为 PDF/图片，用于安全审计和事件复盘
3. **辅助培训**：新分析师可以通过查看推理链快速学习资深分析师的研判思路
4. **人工修正**：分析师可在推理链任意阶段进行修正，修正后的推理链会反馈给策略自演化系统

### Q12：多模型智能路由是如何工作的？如何选择模型？

**A**：多模型智能路由（Model Router）根据任务特征自动选择最优模型：
- **任务类型**：简单分类任务使用轻量模型，复杂研判使用 GPT-4o 或 Claude
- **置信度阈值**：低置信度任务自动升级到更强模型重新分析
- **延迟要求**：实时性要求高的任务路由到低延迟模型
- **成本控制**：在预算范围内自动平衡模型选择

系统默认配置已覆盖大部分场景，高级用户可在 **系统设置 → AI 配置** 中自定义路由规则。

### Q13：策略自演化和之前的反馈学习有什么区别？

**A**：策略自演化是反馈学习的升级版本：
- **反馈学习**：基于人工反馈调整 AI 研判模型的权重，属于被动学习
- **策略自演化**：基于 Q-Learning 强化学习算法，主动分析历史处置效果，自动优化置信度阈值、处置动作优先级、审批规则等策略参数
- **核心差异**：策略自演化不仅学习"判断是否正确"，还学习"如何更有效地处置"，实现从研判到处置的全链路优化

### Q14：多租户隔离是如何实现的？数据安全有保障吗？

**A**：v3.0 支持三种数据隔离级别：
1. **Schema 隔离**：每个租户拥有独立的数据库 Schema，数据完全物理隔离，适用于金融、医疗等强合规行业
2. **RLS 行级隔离**：同一表内通过 `tenant_id` 字段实现行级隔离，适用于中小企业
3. **混合模式**：核心数据 Schema 隔离 + 共享数据 RLS，适用于大型 MSP 服务商

所有隔离方案均经过安全审计，满足 SOC 2 Type II 和等保 2.0 三级要求。

### Q15：WebSocket 实时更新需要额外配置吗？

**A**：WebSocket 默认开启，无需额外配置。系统自动处理以下事项：
- 前端页面加载时自动建立 WebSocket 连接
- 连接断开时自动重连（指数退避策略）
- 连接状态在顶部导航栏实时显示（🟢 已连接 / 🟡 重连中 / 🔴 已断开）
- 支持消息压缩，减少带宽占用

如需自定义配置（如心跳间隔、重连策略），可在 **系统设置 → WebSocket 配置** 中调整。

### Q16：如何从 v2.x 升级到 v3.0？升级会丢失数据吗？

**A**：升级步骤详见 [7.6 升级指南](#76-升级指南v2x--v30)。升级过程不会丢失数据，但建议：
1. 升级前完整备份数据库
2. 在测试环境先验证升级流程
3. 选择业务低峰期进行升级
4. 升级后按照验证清单逐项检查

如有任何升级问题，请联系技术支持：support@secmind.com
